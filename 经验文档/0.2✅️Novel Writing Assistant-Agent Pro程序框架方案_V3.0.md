
**6维度评分权重**：
- 字数: 10%

- 风格: 25%

- 世界观: 20%

2. 评分阈值0.8（低于则继续迭代）
3. 迭代上限5次
4. 6维度评分权重不可修改
5. 上下文记忆前5章

**关键修复记录**：

### 2.2 核心设计原则

#### 2.2.1 微内核原则

**定义**：核心系统仅提供最基本的服务（插件调度、事件分发、依赖管理），所有业务功能通过插件实现。

**核心服务清单**：
| 服务 | 说明 | 接口 |
|------|------|------|
| EventBus | 插件间通信 | `subscribe()`, `publish()`, `publish_sync()` |
| PluginRegistry | 插件管理 | `register()`, `unregister()`, `get_plugin()` |
| ServiceLocator | 依赖注入 | `register_service()`, `get_service()` |
| ConfigManager | 配置管理 | `get_config()`, `set_config()` |

**非核心服务（插件化）**：
- 大纲解析、风格学习、人物管理、世界观解析
- 上下文构建、迭代生成、加权验证
- 热榜功能、统计分析

#### 2.2.2 榫卯式集成原则

**核心理念**：插件与核心通过标准接口"咬合"，接口定义严格，实现灵活。

**接口设计原则**：
1. **最小化接口**：只定义必要的方法，避免过度设计
2. **向后兼容**：新增方法使用默认实现，不破坏现有插件
3. **显式依赖**：插件依赖在清单中明确声明
4. **清晰边界**：插件不直接访问核心内部状态

**示例**：
```python
class BasePlugin(ABC):
    """插件基类 - 榫卯接口"""

    @classmethod
    @abstractmethod
    def get_metadata(cls) -> PluginMetadata:
        """获取插件元数据（榫头）"""
        pass

    @abstractmethod
    def initialize(self, context: PluginContext) -> bool:
        """初始化插件（卯槽）"""
        pass

    @abstractmethod
    def execute(self, *args, **kwargs) -> Any:
        """执行插件核心功能"""
        pass
```

#### 2.2.3 事件驱动原则

**核心理念**：插件间通过事件总线通信，解耦依赖。

**事件类型**：
| 事件类型 | 说明 | 典型场景 |
|----------|------|----------|
| 系统事件 | 插件加载/卸载 | `plugin.loaded`, `plugin.unloaded` |
| 生成事件 | 生成流程 | `generation.started`, `generation.completed` |
| 验证事件 | 质量验证 | `validation.completed` |
| 用户事件 | 用户操作 | `user.generation_requested` |

#### 2.2.4 可观测性原则

**三大支柱**：
1. **结构化日志**：JSON格式 + 追踪ID + 上下文注入
2. **指标采集**：计数器、仪表盘、直方图、计时器
3. **健康检查**：组件级健康状态 + 响应时间

## 四、微内核核心引擎

### 4.1 事件总线 (EventBus)

**核心职责**：插件间发布-订阅通信

**设计要点**：
- 同步/异步双模式
- 支持事件过滤
- 自动追踪ID注入

**关键接口**：
```python
class EventBus:
    def subscribe(self, event_type: str, handler: Callable) -> str:
        """订阅事件，返回订阅ID"""

    def unsubscribe(self, subscription_id: str) -> bool:
        """取消订阅"""

    def publish(self, event_type: str, data: Any, source: str = None):
        """发布事件（异步）"""

    def publish_sync(self, event_type: str, data: Any, timeout: float = 5.0) -> List[Any]:
        """同步发布，等待所有处理器返回结果"""
```

**实现增强**（后端架构师版本）：
- 重试机制（最多3次）
- 死信队列（最大1000条）
- 超时控制（默认5秒）

### 4.2 插件注册表 (PluginRegistry)

**核心职责**：管理所有已加载的插件

**设计要点**：
- 支持按类型/ID查询
- 插槽绑定机制
- 状态管理（LOADED/ACTIVE/ERROR）

**关键接口**：
```python
class PluginRegistry:
    def register(self, plugin: BasePlugin) -> bool:
        """注册插件"""

    def unregister(self, plugin_id: str) -> bool:
        """注销插件"""

    def get_plugin(self, plugin_id: str) -> Optional[BasePlugin]:
        """获取插件实例"""

    def get_plugins_by_type(self, plugin_type: PluginType) -> List[BasePlugin]:
        """按类型获取插件"""

    def get_slot_plugin(self, slot_id: str) -> Optional[BasePlugin]:
        """获取插槽当前绑定的插件"""
```

### 4.3 服务定位器 (ServiceLocator)

**核心职责**：提供核心服务的依赖注入

**设计要点**：
- 单例模式
- 线程安全
- 支持循环依赖检测

**关键接口**：
```python
class ServiceLocator:
    def register_service(self, service_type: Type, instance: Any):
        """注册服务实例"""

    def get_service(self, service_type: Type) -> Any:
        """获取服务实例"""

    def has_service(self, service_type: Type) -> bool:
        """检查服务是否存在"""
```

### 4.4 配置管理器 (ConfigManager)

**核心职责**：统一配置管理

**设计要点**：

### 5.2 插件接口标准

#### 5.2.1 基础插件接口 (BasePlugin)

```python
@dataclass
class PluginMetadata:
    """插件元数据"""
    id: str                          # 唯一标识符（如：outline-parser-v3）
    name: str                        # 显示名称
    version: str                     # 版本号（语义化版本）
    description: str                 # 描述
    author: str                      # 作者
    plugin_type: PluginType          # 插件类型
    api_version: str = "1.0"         # API版本
    priority: int = 100              # 加载优先级（越小越先）
    enabled: bool = True             # 是否启用
    dependencies: List[str] = field(default_factory=list)  # 依赖插件ID列表
    conflicts: List[str] = field(default_factory=list)     # 冲突插件ID列表
    permissions: List[str] = field(default_factory=list)   # 所需权限
    min_platform_version: str = "6.0.0"  # 最低平台版本

class BasePlugin(ABC):
    """插件基类 - 所有插件必须继承"""

    @classmethod
    @abstractmethod
    def get_metadata(cls) -> PluginMetadata:
        """获取插件元数据"""
        pass

    @abstractmethod
    def initialize(self, context: PluginContext) -> bool:
        """初始化插件"""
        pass

    @abstractmethod
    def execute(self, *args, **kwargs) -> Any:
        """执行插件核心功能"""
        pass

    @abstractmethod
    def shutdown(self) -> bool:
        """关闭插件"""
        pass
```

#### 5.2.2 专用插件接口

**AnalyzerPlugin示例**：
```python
class AnalyzerPlugin(BasePlugin):
    """分析器插件 - 分析文本内容"""

    @abstractmethod
    def analyze(self, content: str, options: Dict[str, Any] = None) -> Dict[str, Any]:
        """分析内容"""
        pass

    @abstractmethod
    def get_supported_formats(self) -> List[str]:
        """获取支持的输入格式"""
        pass

    @abstractmethod
    def get_analysis_types(self) -> List[str]:
        """获取支持的分析类型"""
        pass
```

**GeneratorPlugin示例**：
```python
class GeneratorPlugin(BasePlugin):
    """生成器插件 - 生成内容"""

    @abstractmethod
    def generate(self, context: Dict[str, Any], params: Dict[str, Any]) -> Dict[str, Any]:
        """生成内容"""
        pass

    @abstractmethod
    def validate_context(self, context: Dict[str, Any]) -> Tuple[bool, List[str]]:
        """验证上下文是否满足生成要求"""
        pass

    @abstractmethod
    def get_generation_options(self) -> Dict[str, Any]:
        """获取生成选项定义"""
        pass
```

### 5.3 插件加载机制

#### 5.3.1 加载策略对比

| 机制 | 复杂度 | 性能 | 隔离性 | 热插拔 | 适用场景 |
|------|--------|------|--------|--------|----------|
| 动态导入 | 低 | 高 | 低 | 部分 | Python插件（推荐） |
| 动态库 | 中 | 高 | 中 | 支持 | C/C++扩展 |
| 进程隔离 | 高 | 中 | 高 | 支持 | 高隔离要求 |
| RPC调用 | 高 | 中 | 高 | 支持 | 分布式部署 |

**架构师决策**：采用**动态导入**作为主要机制（实现简单，性能高），仅在特殊场景使用进程隔离。

#### 5.3.2 加载流程

```

    ↓
    扫描插件目录
    解析plugin.json
    检查依赖关系
    ↓
加载阶段 (Loading)
    ↓
    动态导入模块
    验证接口兼容
    创建插件实例
    ↓
初始化阶段 (Initialization)
    ↓
    调用initialize()
    注册到Registry
    绑定事件监听
    ↓
完成
```

#### 5.3.3 热插拔机制

**实现方式**：使用watchdog监控插件目录变化

**关键特性**：
- 防抖重载（1秒内多次修改只重载一次）
- 自动依赖解析（拓扑排序）
- 资源清理（事件监听器、模块缓存）

**架构师警告**：热插拔主要用于开发环境，生产环境建议使用完整重启以确保稳定性。

## 六、Agent深度集成

### 6.1 MasterAgent架构

**核心职责**：协调多个专家Agent协作完成复杂任务

**专家Agent清单**：
| Agent | 职责 | 关键能力 |
|-------|------|----------|
| Thinker | 深度思考推理 | CoT（思维链）、ToT（思维树） |
| Optimizer | 迭代优化 | 反馈驱动、自我修正 |
| Validator | 质量验证 | 多维度评分、规则检查 |
| Planner | 任务规划 | 目标分解、依赖分析 |

### 6.2 Agent协作模式

**模式1：链式协作**
```
Planner → Thinker → Optimizer → Validator → Planner（循环）
```

**模式2：树状协作**
```
Planner
    ├── Thinker (章节大纲)
    ├── Thinker (角色设计)
    └── Thinker (世界观)
        ↓
    Optimizer
        ↓
    Validator
```

**模式3：网状协作**
```
┌─────────┐
│ Planner │ ←──→ Validator (持续验证)
└────┬────┘
     │
     ├─────── Thinker (并行思考多个方面)
     │
     ↓
  Optimizer
```

### 6.3 Agent与插件系统对接

**对接原则**：
1. Agent能力包装为插件（AIPlugin）
2. Agent调用通过PluginManager
3. Agent状态持久化到SQLite

**示例**：
```python
# ThinkerAgent包装为插件
class ThinkerAgentPlugin(AIPlugin):
    def generate_text(self, prompt: str, context: Dict[str, Any]) -> str:
        # 调用MasterAgent的Thinker
        result = self._master_agent.think(prompt, context)
        return result

# 在GeneratorPlugin中使用
class NovelGeneratorPlugin(GeneratorPlugin):
    def generate(self, context, params):
        # 通过PluginManager获取Thinker
        thinker = self._context.service_locator.get_service(ThinkerAgentPlugin)
        outline = thinker.generate_text(outline_prompt, context)
        # ...
```

## 十四、关键技术决策

### 14.1 为什么选择SQLite作为持久化层？

**决策**：使用SQLite作为本地嵌入式数据库

**理由**：
1. **零配置部署**：无需额外安装和配置
2. **单文件存储**：便于备份和迁移
3. **足够性能**：配合WAL模式，性能满足需求
4. **成熟稳定**：全球部署最广的数据库
5. **Python原生支持**：无额外依赖

**备选方案**：PostgreSQL（过于重量级）、MySQL（部署成本高）、文件系统（查询效率低）

**权衡分析**：
- 优势：零配置、单文件、性能足够、成熟稳定
- 劣势：不适合高并发、分布式能力弱
- 缓解措施：WAL模式、数据导出工具

### 14.2 为什么使用openai SDK作为统一客户端？

**决策**：使用openai SDK封装所有LLM调用

**理由**：
1. **标准化接口**：大多数LLM提供商（包括DeepSeek）都兼容
2. **减少依赖**：只需安装一个库
3. **统一API**：不同模型使用相同调用方式
4. **成熟稳定**：经过大规模验证

**备选方案**：各自专用SDK（依赖多）、原生HTTP请求（开发成本高）、LangChain（过于重量级）

**权衡分析**：
- 优势：标准化、减少依赖、统一API、成熟稳定
- 劣势：可能无法使用某些模型的专用功能
- 缓解措施：保留扩展点，允许定制逻辑

### 14.3 为什么实现自研向量存储？

**决策**：实现SimpleVectorStore而非使用专业向量数据库

**理由**：
1. **零部署成本**：基于SQLite实现，无需独立部署
2. **足够性能**：单用户桌面应用完全满足需求
3. **无外部依赖**：降低系统复杂度
4. **易于备份**：向量数据与业务数据在同一文件

**备选方案**：Milvus（过于重量级）、Pinecone（付费）、Chroma（仍需额外进程）

**权衡分析**：
- 优势：零部署、足够性能、无依赖、易于备份
- 劣势：不适用于超大规模数据、检索性能不如专业数据库
- 缓解措施：实现基础索引优化，保留接口便于升级

### 14.4 为什么采用熔断器模式？

**决策**：在AI服务调用处添加熔断器保护

**理由**：
1. **防止故障级联**：快速失败，避免系统卡死
2. **自动恢复**：HALF_OPEN状态自动探测服务恢复
3. **提升用户体验**：快速失败优于长时间等待
4. **降低资源消耗**：避免无效重试

**备选方案**：简单重试（无法应对持续故障）、超时控制（无法防止故障级联）、限流（适用场景不同）

**权衡分析**：
- 优势：防止故障级联、自动恢复、提升用户体验
- 劣势：增加系统复杂度，需要配置阈值
- 缓解措施：提供默认配置，支持动态调整

#### 架构相关

**微内核架构**：一种软件架构模式，核心系统仅提供最基本的服务，其他功能通过插件或服务进程实现。

**榫卯式架构**：微内核架构的设计理念，强调组件之间的精准匹配和无缝衔接。

#### 插件系统相关

**插件**：独立开发的软件组件，通过标准接口与核心系统交互，提供特定功能。

**热插拔**：在系统运行时动态加载或卸载插件，无需重启系统。

#### 后端架构相关

**事件总线**：一种发布-订阅模式的消息传递机制，用于插件间解耦通信。

**熔断器**：一种容错模式，用于防止故障级联扩散。当某个服务连续失败达到阈值时，熔断器打开，暂时停止请求该服务。

#### AI工程相关

**LLM客户端**：统一封装的大语言模型调用客户端，支持多种模型和提供商。

**提示词工程**：设计和优化AI模型的提示词，以获得期望的输出结果。

**RAG**：检索增强生成，通过检索相关文档来增强AI模型的回答质量。

**A/B测试**：一种对比测试方法，将用户随机分配到不同组，比较不同方案的效果。
