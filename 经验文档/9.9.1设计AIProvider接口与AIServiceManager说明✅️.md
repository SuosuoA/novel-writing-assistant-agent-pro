## 一、设计目标

### 1.1 核心目标

确保以下三个核心功能能够根据用户设置，**动态切换调用本地大模型或线上API**，且切换后立即生效，无需重启应用：

1. **开始创作**（小说生成流水线，包括迭代生成）
2. **快捷创作**（一键生成世界观、大纲、人设、情节）
3. **续写功能**（基于起始文本续写）

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **统一接口** | 屏蔽底层差异，提供一致的调用接口 |
| **动态切换** | 支持运行时配置变更，无需重启 |
| **线程安全** | 所有操作都是线程安全的 |
| **可扩展性** | 支持新增Provider类型（如Azure、Claude等） |
| **错误隔离** | Provider错误不影响ServiceManager |

## 二、架构设计

### 2.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                   业务层（Business Layer）                   │
│  - 开始创作、快捷创作、续写功能等                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│              服务层（Service Layer）                         │
│  - AIServiceManager：统一管理所有AI调用                      │
│  - ConfigService：配置管理                                   │
│  - EventBus：事件总线                                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│              抽象层（Abstraction Layer）                     │
│  - AIProvider：定义统一的AI能力接口                          │
│  - GenerationConfig：生成配置                                │
│  - GenerationResult：生成结果                                │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ OnlineProvider│   │ LocalProvider │   │ 其他Provider  │
│ (DeepSeek等)  │   │ (Ollama等)    │   │ (预留扩展)    │
└───────────────┘   └───────────────┘   └───────────────┘
```

### 2.2 核心组件职责

| 组件 | 职责 |
|------|------|
| **AIProvider** | 抽象基类，定义所有AI能力接口（生成、分析、估算） |
| **AIServiceManager** | 单例，管理Provider的生命周期，提供统一调用接口 |
| **ConfigService** | 管理配置的读取、写入，发布配置变更事件 |
| **EventBus** | 发布订阅机制，传递配置变更事件 |

#### 3.1.1 核心抽象方法（必须实现）

```python
class AIProvider(ABC):
    """AI能力提供者抽象基类"""
    
    @abstractmethod
    def generate_text(
        self, 
        prompt: str, 
        config: Optional[GenerationConfig] = None,
        **kwargs
    ) -> GenerationResult:
        """生成文本（同步）"""
        pass
    
    @abstractmethod
    def generate_text_stream(
        self, 
        prompt: str,
        callback: Callable[[str], None],
        config: Optional[GenerationConfig] = None,
        **kwargs
    ) -> GenerationResult:
        """流式生成文本（异步回调）"""
        pass
    
    @abstractmethod
    def analyze_text(
        self, 
        text: str, 
        analysis_type: str,
        config: Optional[GenerationConfig] = None,
        **kwargs
    ) -> Dict[str, Any]:
        """分析文本（结构化输出）"""
        pass
    
    @abstractmethod
    def estimate_tokens(self, text: str) -> int:
        """估算token数"""
        pass
    
    @abstractmethod
    def get_model_info(self) -> AIModelInfo:
        """获取模型信息"""
        pass
    
    @abstractmethod
    def is_available(self) -> bool:
        """检查服务是否可用"""
        pass
```

## 四、AIServiceManager设计

### 4.1 单例模式

```python
class AIServiceManager:
    """AI服务管理器（单例）"""
    
    _instance = None
    _lock = threading.Lock()
    
    def __new__(cls):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
                    cls._instance._initialized = False
        return cls._instance
```

### 4.2 核心职责

| 职责 | 说明 |
|------|------|
| **Provider管理** | 根据配置动态创建/切换Provider |
| **配置监听** | 监听配置变更事件，自动重新加载Provider |
| **统一接口** | 对外提供统一的AI调用接口 |
| **线程安全** | 使用锁保护Provider创建和切换 |
| **统计信息** | 记录调用次数、错误率、切换次数 |

### 4.3 配置哈希机制

**目的**: 避免配置未变化时重复创建Provider

```python
def _calculate_config_hash(self, config: Dict[str, Any]) -> str:
    """计算配置哈希值"""
    # 只计算影响Provider选择的配置项
    hash_keys = [
        "service_mode",
        "provider",
        "model",
        "api_key",
        "local_url",
        "temperature",
    ]
    hash_dict = {k: config.get(k, "") for k in hash_keys}
    hash_str = str(sorted(hash_dict.items()))
    return hashlib.md5(hash_str.encode()).hexdigest()
```

### 4.4 Provider切换流程

```
┌─────────────┐
│ 配置变更事件 │
└─────────────┘
       │
       ▼
┌─────────────────┐
│ 计算配置哈希     │
└─────────────────┘
       │
       ▼
┌─────────────────┐     未变化
│ 配置是否变化？   │ ───────────> 跳过
└─────────────────┘
       │ 变化
       ▼
┌─────────────────┐
│ 创建新Provider   │
└─────────────────┘
       │
       ▼
┌─────────────────┐     不可用
│ 验证Provider     │ ───────────> 保留旧Provider
└─────────────────┘
       │ 可用
       ▼
┌─────────────────┐
│ 切换Provider     │
└─────────────────┘
       │
       ▼
┌─────────────────┐
│ 发布切换事件     │
└─────────────────┘
```

def _get_ai_config(self) -> Dict[str, Any]:
    """获取AI配置"""
    config = self._config_service.get_all()
    
    # 兼容旧配置格式（扁平化）
    ai_config = {
        "service_mode": config.get("service_mode", "local"),
        "provider": config.get("provider", "DeepSeek"),
        "model": config.get("model", "deepseek-chat"),
        "api_key": config.get("api_key", ""),
        "base_url": config.get("local_url", "http://localhost:11434/v1"),
        "temperature": config.get("temperature", 0.7),
    }
    
    # 如果有嵌套的local配置，也读取
    local_config = config.get("local", {})
    if local_config:
        ai_config["local"] = local_config
    
    return ai_config

## 六、事件机制设计

### 6.1 事件类型

| 事件类型 | 说明 | 数据 |
|---------|------|------|
| `config.changed` | 配置变更 | `{key, value, source}` |
| `ai.provider.changed` | Provider切换 | `{mode, provider, model, switch_count}` |
| `ai.provider.error` | Provider错误 | `{error, error_type}` |

### 6.2 事件订阅

```python
def _subscribe_config_changes(self):
    """订阅配置变更事件"""
    def on_config_changed(event_data):
        logger.info(f"收到配置变更事件: {event_data}")
        self._reload_provider()
    
    self._event_bus.subscribe("config.changed", on_config_changed)
```

### 6.3 事件发布

```python
# Provider切换成功后发布
self._event_bus.publish(
    "ai.provider.changed",
    {
        "mode": config.get("service_mode", "online"),
        "provider": config.get("provider", "unknown"),
        "model": config.get("model", "unknown"),
        "switch_count": self._provider_switch_count,
    }
)

# Provider创建失败后发布
self._event_bus.publish(
    "ai.provider.error",
    {
        "error": str(e),
        "error_type": type(e).__name__,
    }
)
```

## 十一、关键设计决策

### 11.1 为什么使用单例模式？

**原因**：
1. 全局唯一管理Provider实例
2. 避免重复创建Provider（节省资源）
3. 统一配置监听和事件发布

**权衡**：
- ✅ 优点：全局统一、资源节约
- ❌ 缺点：测试时需要Mock

### 11.2 为什么使用配置哈希？

**原因**：
1. 避免配置未变化时重复创建Provider
2. 减少不必要的Provider切换
3. 提升性能（Provider创建可能耗时）

**权衡**：
- ✅ 优点：性能优化
- ❌ 缺点：需要计算哈希（轻微开销）

### 11.3 为什么延迟导入Provider？

```python
def _create_online_provider(self, config: Dict[str, Any]) -> AIProvider:
    from .online_provider import OnlineProvider  # 延迟导入
    return OnlineProvider(config)
```

**原因**：
1. 避免循环依赖
2. 支持按需加载（减少启动时间）
3. Provider未实现时也能正常启动

**权衡**：
- ✅ 优点：依赖解耦、启动优化
- ❌ 缺点：导入错误延迟发现
