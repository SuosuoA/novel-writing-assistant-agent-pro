## 二、Schema设计详解

### 2.1 核心字段定义

#### 必填字段（Required）

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `knowledge_id` | string | 知识点唯一标识 | `scifi-physics-001` |
| `category` | enum | 知识库分类 | `scifi`/`xuanhuan`/`history`/`general` |
| `domain` | string | 知识领域 | `physics`/`religion`/`mythology` |
| `title` | string | 知识点标题 | `时间膨胀效应` |
| `content` | string | 知识点详细内容（支持Markdown） | `根据狭义相对论...` |
| `keywords` | array | 关键词列表（2-10个） | `["相对论", "时间", "光速"]` |
| `created_at` | datetime | 创建时间 | `2026-03-25T19:00:00+08:00` |
| `updated_at` | datetime | 更新时间 | `2026-03-25T19:00:00+08:00` |

#### 可选字段（Optional）

| 字段 | 类型 | 说明 | 默认值 |
|------|------|------|--------|
| `embedding` | array | 向量嵌入（OpenAI 1536维 / 本地 384维） | `null` |
| `references` | array | 参考来源 | `[]` |
| `difficulty` | enum | 知识难度 | `intermediate` |
| `tags` | array | 自定义标签 | `[]` |
| `metadata` | object | 元数据 | `{}` |

### 2.2 分类知识库设计（ADR-003）

**双层知识库架构**:

#### L1: 分类知识库（按题材隔离）

| 分类 | 标识 | 知识领域 | 目标数量 |
|------|------|----------|----------|
| **科幻** | `scifi` | 物理学、化学、生物学、天文学 | ≥500条 |
| **玄幻** | `xuanhuan` | 宗教学、神话学、道教、佛教 | ≥300条 |
| **历史** | `history` | 历史事件、人物传记、朝代更替 | ≥200条 |

#### L2: 通用知识库（跨题材复用）

| 分类 | 标识 | 知识领域 | 目标数量 |
|------|------|----------|----------|
| **通用** | `general` | 哲学、逻辑学、语言学、修辞学 | ≥200条 |

**设计理由**（ADR-003）:
1. **精准召回**: 分类知识库按题材隔离，减少跨题材噪声
2. **效率优化**: 只加载当前题材的知识库，节省内存
3. **扩展灵活**: 新增题材只需添加新分类，不影响现有库
4. **通用共享**: 通用知识库（哲学/逻辑）跨题材复用

---

### 2.3 knowledge_id命名规范

**格式**: `{category}-{domain}-{序号}`

**规则**:
- category: 小写英文（scifi/xuanhuan/history/general）
- domain: 小写英文（physics/chemistry/biology等）
- 序号: 3位数字（001-999）

**示例**:
- `scifi-physics-001`: 科幻-物理学-第1条
- `xuanhuan-religion-015`: 玄幻-宗教学-第15条
- `history-events-203`: 历史-历史事件-第203条
- `general-philosophy-001`: 通用-哲学-第1条

### 2.4 embedding字段说明

**向量嵌入生成方式**:

#### 方式1: OpenAI Embedding API

```python
from openai import OpenAI

client = OpenAI(api_key="sk-...")

response = client.embeddings.create(
    model="text-embedding-3-small",
    input="知识点内容文本"
)

embedding = response.data[0].embedding  # 1536维向量
```

**特点**:
- 维度: 1536维（text-embedding-3-small）或3072维（text-embedding-3-large）
- 成本: $0.02/1M tokens（small）/ $0.13/1M tokens（large）
- 优势: 高质量、多语言支持

#### 方式2: 本地模型

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('all-MiniLM-L6-v2')

embedding = model.encode("知识点内容文本")  # 384维向量
```

**特点**:
- 维度: 384维（all-MiniLM-L6-v2）
- 成本: 免费
- 优势: 离线运行、速度快

**Schema支持**: 两种方式生成的embedding都符合Schema定义，维度自动检测。

## 六、与LanceDB向量存储集成

### 6.1 知识点向量编码流程

```
用户添加知识点（JSON格式）
    ↓
JSON Schema验证（必填字段、枚举值、格式）
    ↓
提取content字段（知识点内容）
    ↓
调用Embedding API生成向量
    ↓
存入LanceDB向量库（knowledge_vectors表）
    ↓
持久化JSON文件（knowledge/scifi/physics/001.json）
```

### 6.2 知识点检索流程

```
用户创作章节（如科幻小说）
    ↓
提取章节关键词（"飞船"、"超光速"、"虫洞"）
    ↓
识别题材（科幻）→ 加载分类知识库（物理/化学/生物）
    ↓
向量检索召回 top-10 相关知识点
    ↓
注入ValidationScores评分维度
    ↓
评分反馈：检测知识一致性（如"超光速是否违反相对论"）
```

## 七、Schema设计原则

### 7.1 OpenClaw文件即真相原则

**核心原则**:
1. **文件即真相源**: JSON文件本身是唯一真相，向量索引仅用于加速检索
2. **可读性优先**: Schema设计优先考虑人类可读性和可维护性
3. **可编辑性**: 用户可直接编辑JSON文件修改知识点
4. **版本控制友好**: JSON格式便于Git版本控制

### 7.2 扩展性原则

**扩展点**:
1. **新增分类**: 在`category`枚举中添加新分类
2. **新增领域**: `domain`字段为自由文本，支持任意领域
3. **自定义标签**: `tags`字段支持用户自定义标签
4. **元数据扩展**: `metadata`字段支持任意键值对

### 7.3 简洁性原则

**必填字段最小化**: 只保留核心字段为必填，其他字段可选

**必填字段**（8个）:
- knowledge_id、category、domain、title、content、keywords、created_at、updated_at

**可选字段**（5个）:
- embedding、references、difficulty、tags、metadata
