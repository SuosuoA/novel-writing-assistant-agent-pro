# 13. 知识库与创作灵活联动方案

> **文档版本**: V1.0
> **创建日期**: 2026-03-27
> **执行者**: 后端架构师
> **目的**: 实现知识库与小说创作的灵活联动，支持隐式引用和显式引用

---

## 一、问题背景

### 1.1 用户需求

知识库与小说创作的联动需要非常灵活，不一定要生搬硬套出现在剧情中：

1. **隐式引用**：物理知识可以是侦探剧的一个手法，或者是一个现象
2. **显式引用**：古诗文可以直接出现在文章中
3. **评分联动**：知识库作为评分项参与评分（如现实题材出现魔法不可行）

### 1.2 当前架构局限

| 局限性 | 影响 | 当前状态 |
|-------|------|---------|
| 知识召回预算偏小 | 仅500 tokens（6.25%），知识影响力有限 | 固定分配，无动态调整 |
| 知识点权重低 | knowledge_reference仅8%权重 | 低于风格(19%)、人设(19%) |
| 冲突检测规则简单 | 基于关键词匹配，语义冲突检测不足 | 依赖LLM增强（未集成） |
| 缺乏引用追溯 | 无法追踪生成内容引用了哪些知识点 | 需要新增机制 |
| 不支持显式标记引用 | 用户无法强制包含特定知识点 | 需要知识点ID标记机制 |

## 二、RAG技术调研总结

### 2.1 知识注入方法对比

| 注入方法 | 核心机制 | 优点 | 缺点 | 适用场景 |
|---------|---------|------|------|---------|
| **直接拼接法** | 检索到的文档片段直接拼接到提示后 | 实现简单、计算开销低 | 容易生硬过渡、风格不一致 | 简单问答、事实查询 |
| **模板化注入法** | 使用预定义模板结构化注入 | 结构清晰、易于控制知识边界 | 灵活性有限 | 专业领域问答 |
| **结构化输入法** | 按逻辑结构（前提-证据-结论）组织 | 支持复杂推理、逻辑连贯 | 需要预处理 | 学术写作、论证型内容 |
| **参数化注入法** | 文档编码为轻量级参数（LoRA）注入 | 降低上下文开销、对冲突更鲁棒 | 细粒度证据整合能力有限 | 长文档处理 |
| **混合注入法** | 结合Token级上下文与参数化知识 | 综合两者优势、性能最佳 | 实现复杂、成本较高 | 复杂推理、高精度要求 |

### 2.2 关键架构挑战

1. **生成与检索的冲突**：LLM本质是生成模型，检索信息注入改变了原始生成机制
2. **注意力机制过载**：需在不连贯的信息块之间管理注意力分布
3. **知识冲突消解**：模型预训练知识、检索结果、不同来源之间存在矛盾
4. **来源可靠性处理**：检索内容可能包含过时、错误或低质量信息

### 2.3 创意写作特殊需求

**温度参数调控**：
- **创意任务推荐**：温度0.7–1.0，配合`top_p=0.9`
- **事实性任务推荐**：温度0.1–0.3，确保输出稳定
- **平衡场景**：温度0.5–0.7，使用`top_k`或`top_p`约束

**隐式引用策略**：
- 中等温度（0.6–0.8）保持创意性
- 通过提示约束确保知识准确性
- 模板示例："基于以下背景知识，创作一段体现其原理的情节。不要直接引用原文，而是将知识融入叙事中。"

**显式引用策略**：
- RAG Citation机制：自动生成引用，支持可信度验证
- 幻觉检测：检测生成内容中源文档未包含的实体信息

### 4.2 关键瓶颈识别

**性能瓶颈**：
1. 向量检索延迟：P99目标<100ms，但多源召回叠加可能超时
2. Token预算分配：固定分配缺乏动态调整，知识召回预算偏小
3. 冲突检测规则：基于关键词匹配，检测深度不足

**数据流瓶颈**：
1. JSON→向量库同步：批量导入时同步延迟
2. 上下文摘要构建：token截断可能丢失关键信息
3. 评分反馈循环：缺乏知识点引用的细粒度追踪

### 5.3 引用策略选择器

**文件**: `core/knowledge_reference_selector.py`

```python
class KnowledgeReferenceSelector:
    """知识点引用策略选择器"""
    
    def select_reference_strategy(
        self,
        knowledge: KnowledgePoint,
        context: Dict[str, Any]
    ) -> Dict[str, Any]:
        """
        选择知识点的引用策略
        
        Args:
            knowledge: 知识点
            context: 上下文（题材、剧情、风格）
        
        Returns:
            {
                "strategy": "explicit" | "implicit" | "constraint",
                "guidance": "如何引用",
                "priority": 0-1
            }
        """
        # 显式引用条件
        if self._is_explicit_candidate(knowledge, context):
            return {
                "strategy": "explicit",
                "guidance": knowledge.reference_guidance.explicit_usage,
                "priority": knowledge.importance_score
            }
        
        # 隐式引用条件
        if self._is_implicit_candidate(knowledge, context):
            return {
                "strategy": "implicit",
                "guidance": knowledge.reference_guidance.implicit_usage,
                "priority": knowledge.importance_score * 0.8
            }
        
        # 约束检查
        return {
            "strategy": "constraint",
            "guidance": knowledge.reference_guidance.constraint_rules,
            "priority": 1.0
        }
    
    def _is_explicit_candidate(self, knowledge, context):
        """判断是否适合显式引用"""
        # 古诗文、名人名言、历史典故
        explicit_types = ["poetry", "quote", "history_event"]
        return (
            knowledge.reference_type == "explicit"
            or knowledge.domain in explicit_types
        )
    
    def _is_implicit_candidate(self, knowledge, context):
        """判断是否适合隐式引用"""
        # 物理、化学、生物原理
        return knowledge.reference_type == "implicit"
```

### 5.5 知识点引用追溯器

**文件**: `core/knowledge_reference_tracker.py`

```python
class KnowledgeReferenceTracker:
    """知识点引用追溯器"""
    
    def track_references(
        self, 
        content: str, 
        recalled_knowledge: List[RecallResult]
    ) -> ReferenceReport:
        """
        追溯生成内容引用的知识点
        
        Returns:
            ReferenceReport {
                total_references: int,
                explicit_references: List,  # 显式引用
                implicit_references: List,  # 隐式引用
                quality_score: float
            }
        """
        explicit_refs = []
        implicit_refs = []
        
        for kp in recalled_knowledge:
            # 显式引用检测（原文出现）
            if self._detect_explicit_reference(content, kp):
                explicit_refs.append(KnowledgeReference(
                    knowledge_id=kp.knowledge_id,
                    title=kp.title,
                    match_score=1.0,
                    reference_type="explicit"
                ))
            
            # 隐式引用检测（语义匹配）
            elif self._detect_implicit_reference(content, kp):
                implicit_refs.append(KnowledgeReference(
                    knowledge_id=kp.knowledge_id,
                    title=kp.title,
                    match_score=self._calculate_match_score(content, kp),
                    reference_type="implicit"
                ))
        
        return ReferenceReport(
            total_references=len(explicit_refs) + len(implicit_refs),
            explicit_references=explicit_refs,
            implicit_references=implicit_refs,
            quality_score=self._calculate_quality(explicit_refs, implicit_refs)
        )
```

```python
class LLMConflictDetector:
    """LLM增强冲突检测器"""
    
    DETECTION_PROMPT = """
你是一位小说创作知识库专家。请检测以下内容是否违反知识点：

【生成内容】
{content}

【相关知识点】
{knowledge_text}

请以JSON格式输出：
{
    "has_conflict": true/false,
    "conflicts": [
        {
            "type": "physics/history/religion",
            "severity": "P0/P1/P2",
            "description": "详细冲突描述",
            "suggested_fix": "具体修复建议"
        }
    ]
}
"""
    
    def detect_with_llm(
        self, 
        content: str, 
        recalled_knowledge: List[RecallResult],
        llm_client
    ) -> List[KnowledgeConflict]:
        """使用LLM检测冲突"""
        knowledge_text = "\n\n".join([
            f"【{kp.title}】{kp.content}" 
            for kp in recalled_knowledge
        ])
        
        prompt = self.DETECTION_PROMPT.format(
            content=content,
            knowledge_text=knowledge_text
        )
        
        response = llm_client.chat.completions.create(
            model=llm_client.model,
            messages=[{"role": "user", "content": prompt}]
        )
        
        result = json.loads(response.choices[0].message.content)
        
        conflicts = []
        for c in result.get("conflicts", []):
            conflicts.append(KnowledgeConflict(
                conflict_type=c["type"],
                severity=c["severity"],
                description=c["description"],
                suggested_fix=c["suggested_fix"]
            ))
        
        return conflicts
```

## 六、评分机制增强

### 6.1 扩展knowledge_reference维度

**当前**: 仅检测是否违反知识库（扣分机制）

**增强**: 多维评分

```python
class KnowledgeReferenceScorer:
    """知识点引用评分器"""
    
    def calculate_score(
        self,
        content: str,
        recalled_knowledge: List[RecallResult],
        reference_report: ReferenceReport
    ) -> Dict[str, float]:
        """
        计算知识点引用评分
        
        Returns:
            {
                "reference_count_score": 0-1,    # 引用数量评分
                "reference_quality_score": 0-1,  # 引用质量评分
                "reference_naturalness": 0-1,    # 引用自然度评分
                "consistency_score": 0-1,        # 知识一致性评分
                "total_score": 0-1               # 总分
            }
        """
        # 引用数量评分（引用越多越好，但有上限）
        ref_count = reference_report.total_references
        count_score = min(ref_count / 5.0, 1.0)  # 5条为满分
        
        # 引用质量评分（显式引用权重更高）
        explicit_weight = len(reference_report.explicit_references) * 1.5
        implicit_weight = len(reference_report.implicit_references) * 1.0
        quality_score = min((explicit_weight + implicit_weight) / 6.0, 1.0)
        
        # 引用自然度评分（检测是否生搬硬套）
        naturalness_score = self._detect_reference_naturalness(
            content, 
            reference_report
        )
        
        # 知识一致性评分（是否违反知识库）
        consistency_score = self._check_consistency(
            content,
            recalled_knowledge
        )
        
        # 加权总分
        total_score = (
            count_score * 0.2
            + quality_score * 0.4
            + naturalness_score * 0.2
            + consistency_score * 0.2
        )
        
        return {
            "reference_count_score": count_score,
            "reference_quality_score": quality_score,
            "reference_naturalness": naturalness_score,
            "consistency_score": consistency_score,
            "total_score": total_score
        }
```

### 6.2 权重调整建议

```
字数: 8%
知识点引用: 8%  ← 权重过低
大纲: 13%
风格: 19%
人设: 19%
世界观: 12%
逆向反馈: 11%
自然度: 10%
```

**建议调整**：
```
字数: 8%
知识点引用: 15%  ← 提升至15%
大纲: 12%
风格: 17%        ← 降低2%
人设: 17%        ← 降低2%
世界观: 12%
逆向反馈: 11%
自然度: 8%        ← 降低2%
```

**理由**：
- 知识库是核心竞争力，应提高权重
- 知识点引用包含多个子维度，需要更高权重体现
- 风格、人设权重过高，可能忽略知识正确性

## 八、风险与应对

### 8.1 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| LLM冲突检测延迟 | 评分反馈循环变慢 | 缓存检测结果、异步检测 |
| Token预算动态调整复杂 | 可能导致其他维度不足 | 兜底机制、用户可配置 |
| 引用追溯准确率不足 | 评分不准确 | 多策略融合、人工校验 |

### 8.2 业务风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 知识点引用过多 | 内容生硬、破坏叙事 | 自然度评分、引用上限 |
| 题材不匹配 | 知识点不适用 | 题材适配验证、过滤机制 |
| 用户不理解引用逻辑 | 评分不可控 | UI提示、用户教育 |
