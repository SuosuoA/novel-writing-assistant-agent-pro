```python
# core/thread_pool_manager.py (新增 +200行)
class ThreadPoolManager:
    """统一线程池管理器 - 单例模式"""
    
    def __init__(self):
        # 8个工作线程（处理异步任务）
        self._executor = ThreadPoolExecutor(max_workers=8)
        # 1个事件循环线程（AsyncIO）
        self._loop_thread: Optional[threading.Thread] = None
        self._loop: Optional[asyncio.AbstractEventLoop] = None
        self._lock = threading.RLock()
    
    def submit_async_safe(self, coro, timeout: float = 300.0,
                          retries: int = 3) -> Future:
        """安全提交异步任务（带重试和超时）"""
        pass
    
    def submit_sync_safe(self, func, *args, 
                         timeout: float = 300.0,
                         retries: int = 3) -> Future:
        """安全提交同步任务（带重试和超时）"""
        pass
    
    def shutdown(self, wait: bool = True, timeout: float = 30.0):
        """优雅关闭（等待任务完成）"""
        pass
```

### 2.2 核心设计原则

#### 2.2.1 微内核原则

核心系统仅提供最基本的服务（插件调度、事件分发、依赖管理），所有业务功能通过插件实现。

**核心服务清单**：

| 服务 | 说明 | 关键接口 |
|------|------|---------|
| EventBus | 插件间通信 | `subscribe()`, `publish()`, `publish_sync()` |
| PluginRegistry | 插件管理 | `register()`, `unregister()`, `get_plugin()` |
| ServiceLocator | 依赖注入 | `register()`, `get()` ⚠️ **V1.10.0修订** |
| ConfigManager | 配置管理 | `get()`, `set()` ⚠️ **V1.10.0修订** |

#### 2.2.2 事件驱动原则

插件间通过事件总线通信，解耦依赖。

**事件类型**：

| 事件类型 | 说明 | 典型场景 |
|----------|------|----------|
| 系统事件 | 插件加载/卸载 | `plugin.loaded`, `plugin.unloaded`, `plugin.error` |
| 生成事件 | 生成流程 | `generation.started`, `generation.progress`, `generation.completed` |
| 验证事件 | 质量验证 | `validation.completed` |
| 用户事件 | 用户操作 | `user.generation_requested` |

### 3.2 插件状态管理

**5状态模型**：

```
UNLOADED → LOADED → ACTIVE → UNLOADING → UNLOADED
              ↓         ↓
            ERROR ←←←←←←
```

| 状态 | 说明 | 可转换状态 |
|------|------|-----------|
| `UNLOADED` | 未加载（初始状态） | → LOADED |
| `LOADED` | 已加载，未激活 | → ACTIVE, ERROR, UNLOADING |
| `ACTIVE` | 已激活，可执行 | → LOADED, ERROR, UNLOADING |
| `ERROR` | 错误状态 | → LOADED, UNLOADING |
| `UNLOADING` | 卸载中 | → 从注册表移除 |

### 6.1 架构设计哲学

**核心原则：文件即真相源（File is the Source of Truth）**

OpenClaw mem9框架建立在纯文本Markdown文件之上：
1. **纯文本驱动**：所有记忆以Markdown格式存储，透明可编辑
2. **不重新训练模型**：通过写入和检索记忆文件来适应，而非改动底层LLM
3. **向量索引非真相源**：文件本身是唯一真相，向量索引仅用于加速检索

```python
class ActiveTask(BaseModel):
    """活跃任务"""
    task_id: str
    task_type: str  # "generation", "validation", "analysis"
    description: str
    started_at: datetime

class TempContext(BaseModel):
    """临时上下文"""
    current_chapter: Optional[int] = None
    current_file: Optional[str] = None
    last_operation: Optional[str] = None
    pending_data: dict = Field(default_factory=dict)

class ErrorState(BaseModel):
    """异常状态"""
    has_error: bool = False
    error_message: Optional[str] = None
    error_time: Optional[datetime] = None
    recovery_hint: Optional[str] = None

class SessionState(BaseModel):
    """会话状态"""
    state_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
    active_task: Optional[ActiveTask] = None
    temp_context: TempContext = Field(default_factory=TempContext)
```

**问题1：知识点验证失败**

- UI详情面板高度只有6行，显示空间不足
- 双击显示逻辑未显示所有可用信息

#### 解决方案

**修复1：数据模型字段类型兼容**
```python
# core/knowledge_manager.py
# 修复前
references: Optional[List[str]] = Field(default=None)

# 修复后
references: Optional[Union[List[str], str]] = Field(default=None)

@field_validator("references", mode="before")
@classmethod
def normalize_references(cls, v):
    """将字符串references转换为列表"""
    if isinstance(v, str):
        return [v]
    return v
```

**修复2：验证规则放宽**
```python
# 修复前：最小长度50字符
if len(v) < 50:
    raise ValueError("知识点内容长度不能少于50字符")

# 修复后：最小长度10字符

```

### 7.2 人设评分低分问题修复（V5.5）

**修复**：

### 7.3 【本章完】标记缺失修复（V5.6）

**问题**：AI可能未生成标记导致整个生成失败

**修复**：
1. `iterative_generator_v2.py`在最终输出时自动智能补充标记
2. `novel_generator_v3_2.py`做双重保险

### 7.4 资源管理与异常处理

**线程池关闭参数兼容性**：
```python
try:
    self._executor.shutdown(wait=True, cancel_futures=True)
except TypeError:
    self._executor.shutdown(wait=True)  # Python 3.8及以下
```

**异常处理原则**：
1. 不吞掉异常，所有异常必须记录日志
2. 日志完整性：异常信息 + 相关对象引用 + 关键参数
3. 关键异常发布到EventBus供监控系统感知
4. 使用`atexit`确保退出时清理资源

**资源泄漏防护**：
- 连接池：超时回收机制 + 优雅关闭 + 状态标记
- 队列：容量限制（`AsyncHandler.DEFAULT_MAX_SIZE = 1000`），满时拒绝新任务

**状态一致性**：
- 关键服务失败立即终止初始化
- 依赖服务不强制初始化
- 初始化状态必须反映实际结果

```python
from pydantic import BaseModel, Field
from typing import List, Optional
from datetime import datetime
from enum import Enum

class ChapterStatus(str, Enum):
    DRAFT = "draft"           # 草稿
    COMPLETED = "completed"   # 已完成
    ANALYZED = "analyzed"     # 已分析

class ChapterData(BaseModel):
    """章节数据模型"""
    chapter_id: str = Field(..., description="章节唯一ID")
    chapter_num: int = Field(..., description="章节序号")
    title: str = Field(..., description="章节标题")
    content: str = Field(..., description="章节正文内容")
    word_count: int = Field(default=0, description="字数")
    status: ChapterStatus = Field(default=ChapterStatus.DRAFT, description="状态")
    created_at: datetime = Field(default_factory=datetime.now)
    updated_at: datetime = Field(default_factory=datetime.now)
    analyzed_at: Optional[datetime] = Field(default=None, description="分析时间")
    
    # 影响范围
    affected_outlines: List[str] = Field(default_factory=list, description="影响的大纲节点ID")
    affected_characters: List[str] = Field(default_factory=list, description="影响的人物ID")
    affected_worldview: List[str] = Field(default_factory=list, description="影响的世界观元素ID")

class FeedbackReport(BaseModel):
    """反馈报告模型"""
    report_id: str = Field(..., description="报告ID")
    chapter_id: str = Field(..., description="关联章节ID")
    generated_at: datetime = Field(default_factory=datetime.now)
    
    # 四维度检测结果

    character_score: float = Field(ge=0, le=1, description="人设得分")
    style_score: float = Field(ge=0, le=1, description="风格得分")
    
    # 修正建议
    suggestions: List[dict] = Field(default_factory=list, description="修正建议列表")
    
    # 影响范围
    impact_summary: str = Field(default="", description="影响范围摘要")
```

#### 功能概述

续写功能是根据已有原文内容，智能分析并生成后续内容。

**核心流程**：
1. 输入/导入原文内容
2. 设置续写参数（字数、方向）
3. 调用LLM生成续写内容
4. 支持多次尝试，选择最佳版本

**续写方向选项**：
- 自然续写、制造冲突、引入新角色、转折情节、高潮推进

#### 使用场景

1. 用户写了一段开头，希望AI继续写下去
2. 用户对当前生成结果不满意，想换一种风格或方向重新生成
3. 用户需要生成多个版本，从中挑选最佳

#### 设计思路

1. **插件化实现**：开发 ContinuationPlugin（继承 GeneratorPlugin），提供续写能力
2. **上下文智能注入**：自动读取当前项目的大纲、人设、世界观作为生成上下文
3. **多版本管理**：前端维护生成历史，支持重新生成和版本选择
4. **与项目集成**：保存的续写内容可作为新章节或追加到已有章节
