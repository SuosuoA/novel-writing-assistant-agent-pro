### 2.2 核心设计原则

#### 2.2.1 微内核原则

**定义**：核心系统仅提供最基本的服务（插件调度、事件分发、依赖管理），所有业务功能通过插件实现。

**核心服务清单**：
| 服务 | 说明 | 接口 |
|------|------|------|
| EventBus | 插件间通信 | `subscribe()`, `publish()`, `publish_sync()` |
| PluginRegistry | 插件管理 | `register()`, `unregister()`, `get_plugin()` |
| ServiceLocator | 依赖注入 | `register_service()`, `get_service()` |
| ConfigManager | 配置管理 | `get_config()`, `set_config()` |

**非核心服务（插件化）**：
- 大纲解析、风格学习、人物管理、世界观解析
- 上下文构建、迭代生成、加权验证
- 热榜功能、统计分析

#### 2.2.2 榫卯式集成原则

**核心理念**：插件与核心通过标准接口"咬合"，接口定义严格，实现灵活。

**接口设计原则**：
1. **最小化接口**：只定义必要的方法，避免过度设计
2. **向后兼容**：新增方法使用默认实现，不破坏现有插件
3. **显式依赖**：插件依赖在清单中明确声明
4. **清晰边界**：插件不直接访问核心内部状态

**示例**：
```python
class BasePlugin(ABC):
    """插件基类 - 榫卯接口"""

    @classmethod
    @abstractmethod
    def get_metadata(cls) -> PluginMetadata:
        """获取插件元数据（榫头）"""
        pass

    @abstractmethod
    def initialize(self, context: PluginContext) -> bool:
        """初始化插件（卯槽）"""
        pass

    @abstractmethod
    def execute(self, *args, **kwargs) -> Any:
        """执行插件核心功能"""
        pass
```

#### 2.2.3 事件驱动原则

**核心理念**：插件间通过事件总线通信，解耦依赖。

**事件类型**：
| 事件类型 | 说明 | 典型场景 |
|----------|------|----------|
| 系统事件 | 插件加载/卸载 | `plugin.loaded`, `plugin.unloaded` |
| 生成事件 | 生成流程 | `generation.started`, `generation.completed` |
| 验证事件 | 质量验证 | `validation.completed` |
| 用户事件 | 用户操作 | `user.generation_requested` |

#### 2.2.4 可观测性原则

**三大支柱**：
1. **结构化日志**：JSON格式 + 追踪ID + 上下文注入
2. **指标采集**：计数器、仪表盘、直方图、计时器
3. **健康检查**：组件级健康状态 + 响应时间

## 四、微内核核心引擎

### 4.1 事件总线 (EventBus)

**核心职责**：插件间发布-订阅通信

**设计要点**：
- 同步/异步双模式
- 支持事件过滤
- 自动追踪ID注入

**关键接口**：
```python
class EventBus:
    def subscribe(self, event_type: str, handler: Callable) -> str:
        """订阅事件，返回订阅ID"""

    def unsubscribe(self, subscription_id: str) -> bool:
        """取消订阅"""

    def publish(self, event_type: str, data: Any, source: str = None):
        """发布事件（异步）"""

    def publish_sync(self, event_type: str, data: Any, timeout: float = 5.0) -> List[Any]:
        """同步发布，等待所有处理器返回结果"""
```

**实现增强**（后端架构师版本）：
- 重试机制（最多3次）
- 死信队列（最大1000条）
- 超时控制（默认5秒）

### 4.2 插件注册表 (PluginRegistry)

**核心职责**：管理所有已加载的插件

**设计要点**：
- 支持按类型/ID查询
- 插槽绑定机制
- 状态管理（LOADED/ACTIVE/ERROR）

**关键接口**：
```python
class PluginRegistry:
    def register(self, plugin: BasePlugin) -> bool:
        """注册插件"""

    def unregister(self, plugin_id: str) -> bool:
        """注销插件"""

    def get_plugin(self, plugin_id: str) -> Optional[BasePlugin]:
        """获取插件实例"""

    def get_plugins_by_type(self, plugin_type: PluginType) -> List[BasePlugin]:
        """按类型获取插件"""

    def get_slot_plugin(self, slot_id: str) -> Optional[BasePlugin]:
        """获取插槽当前绑定的插件"""
```

### 4.3 服务定位器 (ServiceLocator)

**核心职责**：提供核心服务的依赖注入

**设计要点**：
- 单例模式
- 线程安全
- 支持循环依赖检测

**关键接口**：
```python
class ServiceLocator:
    def register_service(self, service_type: Type, instance: Any):
        """注册服务实例"""

    def get_service(self, service_type: Type) -> Any:
        """获取服务实例"""

    def has_service(self, service_type: Type) -> bool:
        """检查服务是否存在"""
```

### 4.4 配置管理器 (ConfigManager)

**核心职责**：统一配置管理

**设计要点**：

### 5.2 插件接口标准

#### 5.2.1 基础插件接口 (BasePlugin)

```python
@dataclass
class PluginMetadata:
    """插件元数据"""
    id: str                          # 唯一标识符（如：outline-parser-v3）
    name: str                        # 显示名称
    version: str                     # 版本号（语义化版本）
    description: str                 # 描述
    author: str                      # 作者
    plugin_type: PluginType          # 插件类型
    api_version: str = "1.0"         # API版本
    priority: int = 100              # 加载优先级（越小越先）
    enabled: bool = True             # 是否启用
    dependencies: List[str] = field(default_factory=list)  # 依赖插件ID列表
    conflicts: List[str] = field(default_factory=list)     # 冲突插件ID列表
    permissions: List[str] = field(default_factory=list)   # 所需权限
    min_platform_version: str = "6.0.0"  # 最低平台版本

class BasePlugin(ABC):
    """插件基类 - 所有插件必须继承"""

    @classmethod
    @abstractmethod
    def get_metadata(cls) -> PluginMetadata:
        """获取插件元数据"""
        pass

    @abstractmethod
    def initialize(self, context: PluginContext) -> bool:
        """初始化插件"""
        pass

    @abstractmethod
    def execute(self, *args, **kwargs) -> Any:
        """执行插件核心功能"""
        pass

    @abstractmethod
    def shutdown(self) -> bool:
```

### 9.2 统一线程池管理器（ThreadPoolManager）

**核心设计**：

```python
# core/thread_pool_manager.py (新增)
class ThreadPoolManager:
    """统一线程池管理器 - 单例模式
    
    整合：
    - AsyncHandler的ThreadPoolExecutor
    - threading.Thread的创建
    - AsyncIO事件循环
    """
    
    _instance = None
    _lock = threading.RLock()
    
    def __new__(cls):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance
    
    def __init__(self):
        if hasattr(self, '_initialized'):
            return
        
        # 8个工作线程（处理同步任务）
        self._executor = ThreadPoolExecutor(max_workers=8)
        # 1个事件循环线程（处理异步任务）
        self._loop_thread: Optional[threading.Thread] = None
        self._loop: Optional[asyncio.AbstractEventLoop] = None
        # 线程安全锁
        self._lock = threading.RLock()
        # 初始化标记
        self._initialized = True
    
    def start_loop_thread(self):
        """启动AsyncIO事件循环线程"""
        if self._loop_thread is None or not self._loop_thread.is_alive():
            self._loop = asyncio.new_event_loop()
            self._loop_thread = threading.Thread(
                target=self._run_loop,
                daemon=True,
                name="AsyncLoopThread"
            )
            self._loop_thread.start()
    
    def _run_loop(self):
        """运行事件循环"""
        asyncio.set_event_loop(self._loop)
        self._loop.run_forever()
    
    def submit_async_safe(self, coro, timeout: float = 300.0,
                          retries: int = 3) -> asyncio.Future:
        """安全提交异步任务（带重试和超时）"""
        self.start_loop_thread()
        
        async def _execute_with_retry():
            for attempt in range(retries):
                try:
                    return await asyncio.wait_for(coro, timeout=timeout)
                except asyncio.TimeoutError:
                    if attempt == retries - 1:
                        raise
                    await asyncio.sleep(1.0 * (attempt + 1))  # 指数退避
                except Exception as e:
                    if attempt == retries - 1:
                        raise
                    await asyncio.sleep(0.5)
        
        return asyncio.run_coroutine_threadsafe(
            _execute_with_retry(), self._loop
        )
    
    def submit_sync_safe(self, func, *args, 
                         timeout: float = 300.0,
                         retries: int = 3) -> Future:
        """安全提交同步任务（带重试和超时）"""
        def _execute_with_retry():
            for attempt in range(retries):
                try:
                    return func(*args)
                except Exception as e:
                    if attempt == retries - 1:
                        raise
                    time.sleep(0.5 * (attempt + 1))
        
        return self._executor.submit(_execute_with_retry)
    
    def shutdown(self, wait: bool = True, timeout: float = 30.0):
        """优雅关闭"""
        # 关闭事件循环
        if self._loop and self._loop.is_running():
            self._loop.call_soon_threadsafe(self._loop.stop)
        
        # 关闭线程池
        self._executor.shutdown(wait=wait, timeout=timeout)
```

**使用示例**:

```python

# 非阻塞获取结果（带超时）
```

### 9.4 PipelineOrchestrator异步化策略

```python
class NovelGenerationService:
    """小说生成服务 - Service层异步调度"""
    
    def __init__(self):
        self._orchestrator = PipelineOrchestrator()
        self._thread_pool = ThreadPoolManager()
    
    async def generate_chapter_async(self, config: NovelGenerationConfig) -> Dict:
        """异步生成章节 - 不修改Orchestrator"""
        # 使用ThreadPoolManager提交同步任务
        return await self._thread_pool.submit_async_safe(
            self._run_generation(config),
            timeout=300.0,
            retries=3
        )
    
    async def _run_generation(self, config: NovelGenerationConfig):
        """在独立线程中运行同步生成"""
        return self._orchestrator.generate_chapter(config)
    
    def generate_chapter(self, config: NovelGenerationConfig) -> Dict:
        """同步生成章节 - 保留向后兼容"""
        return self._orchestrator.generate_chapter(config)
```

```python
    """带降级的异步执行"""
    try:
        # 检查特性开关
        if not self._config.get("async_enabled", True):
            self._logger.info("特性开关关闭，降级到同步")
            return fallback_func()
        
        # 尝试异步执行
        manager = ThreadPoolManager()
        future = manager.submit_async_safe(
            coro, timeout=300.0, retries=3
        )
        return future.result(timeout=1.0)
    
    except Exception as e:
        self._logger.warning(f"异步执行失败，降级到同步: {e}")
        return fallback_func()
```

## 十五、关键技术决策

### 17.1 为什么选择SQLite作为持久化层？

**决策**：使用SQLite作为本地嵌入式数据库

**理由**：
1. **零配置部署**：无需额外安装和配置
2. **单文件存储**：便于备份和迁移
3. **足够性能**：配合WAL模式，性能满足需求
4. **成熟稳定**：全球部署最广的数据库
5. **Python原生支持**：无额外依赖

**备选方案**：PostgreSQL（过于重量级）、MySQL（部署成本高）、文件系统（查询效率低）

**权衡分析**：
- 优势：零配置、单文件、性能足够、成熟稳定
- 劣势：不适合高并发、分布式能力弱
- 缓解措施：WAL模式、数据导出工具

### 17.2 为什么使用openai SDK作为统一客户端？

**决策**：使用openai SDK封装所有LLM调用

**理由**：
1. **标准化接口**：大多数LLM提供商（包括DeepSeek）都兼容
2. **减少依赖**：只需安装一个库
3. **统一API**：不同模型使用相同调用方式
4. **成熟稳定**：经过大规模验证

**备选方案**：各自专用SDK（依赖多）、原生HTTP请求（开发成本高）、LangChain（过于重量级）

**权衡分析**：
- 优势：标准化、减少依赖、统一API、成熟稳定
- 劣势：可能无法使用某些模型的专用功能
- 缓解措施：保留扩展点，允许定制逻辑

### 16.3 为什么实现自研向量存储？

**决策**：实现SimpleVectorStore而非使用专业向量数据库

**理由**：
1. **零部署成本**：基于SQLite实现，无需独立部署
2. **足够性能**：单用户桌面应用完全满足需求
3. **无外部依赖**：降低系统复杂度
4. **易于备份**：向量数据与业务数据在同一文件

**备选方案**：Milvus（过于重量级）、Pinecone（付费）、Chroma（仍需额外进程）

**权衡分析**：
- 优势：零部署、足够性能、无依赖、易于备份
- 劣势：不适用于超大规模数据、检索性能不如专业数据库
- 缓解措施：实现基础索引优化，保留接口便于升级

### 15.4 为什么采用熔断器模式？

**决策**：在AI服务调用处添加熔断器保护

**理由**：
1. **防止故障级联**：快速失败，避免系统卡死
2. **自动恢复**：HALF_OPEN状态自动探测服务恢复
3. **提升用户体验**：快速失败优于长时间等待
4. **降低资源消耗**：避免无效重试

**备选方案**：简单重试（无法应对持续故障）、超时控制（无法防止故障级联）、限流（适用场景不同）

**权衡分析**：
- 优势：防止故障级联、自动恢复、提升用户体验
- 劣势：增加系统复杂度，需要配置阈值
- 缓解措施：提供默认配置，支持动态调整
