## 一、功能概述

### 1.1 什么是逆向反馈

逆向反馈是Novel Writing Assistant-Agent Pro的核心功能之一，用于分析已生成的小说章节内容与项目设定的一致性。它能够：

1. **检测设定冲突**：识别章节内容与大纲、人物设定、世界观之间的矛盾
2. **生成修正建议**：为每个冲突提供可执行的修正方案
3. **自动评分反馈**：将一致性检查结果纳入章节质量评分体系

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| **设定一致性保障** | 确保后续生成内容与已有章节保持一致 |
| **冲突自动检测** | 识别角色行为、情节发展、世界观设定的矛盾 |
| **智能修正建议** | 为每个冲突提供可执行的修正方案 |
| **质量闭环反馈** | 影响章节生成迭代流程，确保输出质量 |

### 1.3 功能入口

- **位置**：工作台 → 逆向反馈
- **快捷键**：无
- **触发方式**：手动运行分析 / 章节生成后自动触发

## 二、快速开始

### 2.1 基本使用流程

```
上传章节 → 配置分析参数 → 运行分析 → 查看结果 → 应用修正
```

### 2.2 操作步骤

#### 步骤1：上传章节

支持三种上传方式：

| 方式 | 操作方法 | 适用场景 |
|------|----------|----------|
| **上传文件** | 点击"上传文件"按钮，选择TXT/DOCX文件 | 单个或少量章节 |
| **批量上传** | 点击"批量上传"按钮，选择文件夹 | 大量章节文件 |
| **粘贴文本** | 点击"粘贴文本"按钮，在文本框中粘贴内容 | 快速测试 |

#### 步骤2：标记章节状态

- 右键点击章节，选择"标记为完成"或"标记为未完成"
- 或点击"全部标记完成"批量处理

#### 步骤3：配置分析参数

**分析维度**（可多选）：
- ✓ 一致性检查（章节间逻辑一致性）
- ✓ 逻辑漏洞检测（情节合理性）
- ✓ 人设偏离检测（角色行为一致性）
- ✓ 风格匹配度（写作风格一致性）
- ✓ 世界观冲突（设定一致性）

**分析范围**（单选）：
- ○ 仅分析选中章节
- ○ 分析所有已完成章节
- ○ 分析所有章节

#### 步骤4：运行分析

点击"运行分析"按钮，等待分析完成。

#### 步骤5：查看结果

在冲突列表中选择问题，查看详细的修正建议。

#### 步骤6：应用修正

点击"应用修正"按钮，将修正建议应用到当前项目设定。

## 四、冲突类型说明

### 4.1 人物冲突（Character）

**定义**：章节中角色行为与人物设定不一致。

**常见场景**：
- 角色能力超出设定范围
- 角色性格表现与设定矛盾
- 角色关系发展不符合设定

**示例**：
```
冲突描述：角色'林逸'在第3章中使用了'御剑术'，但人物设定中其能力仅为'基础剑法'
修正建议：将人物设定中的能力修改为'御剑术'，或修改第3章内容
严重程度：高
```

### 4.2 大纲冲突（Outline）

**定义**：章节情节发展与大纲设定不一致。

**常见场景**：
- 章节内容偏离大纲主线
- 章节顺序与大纲不符
- 关键情节遗漏或错误

**示例**：
```
冲突描述：第5章内容涉及'宗门大战'，但大纲中第5章应为'修炼突破'
修正建议：调整第5章内容或更新大纲规划
严重程度：中
```

### 4.3 世界观冲突（Worldview）

**定义**：章节内容违反世界观设定规则。

**常见场景**：
- 设定规则被违反
- 时间线矛盾
- 地理/环境描述不一致

**示例**：
```
冲突描述：章节中角色使用'瞬移'能力，但世界观设定中'瞬移'为禁术
修正建议：修改为'疾行'或'御剑飞行'，或更新世界观设定
严重程度：高
```

## 五、评分机制

### 5.1 评分公式

逆向反馈分析结果转化为一致性评分（0-1）：

```
基础分 = 1.0（无冲突）
惩罚值 = 高优先级冲突数 × 0.20
        + 中优先级冲突数 × 0.10
        + 低优先级冲突数 × 0.05

最终评分 = max(0.4, 基础分 - 惩罚值)
```

### 5.2 评分示例

| 冲突情况 | 评分 |
|----------|------|
| 无冲突 | 1.0 |
| 1个低优先级 | 0.95 |
| 1个中优先级 | 0.90 |
| 1个高优先级 | 0.80 |
| 2个高优先级 + 1个中优先级 | 0.50 |
| 大量冲突（保底） | 0.40 |

### 5.3 对生成流程的影响

一致性评分作为额外加权因子影响章节生成迭代：

```python
if context_consistency < 0.7:
    penalty_factor = max(0.7, context_consistency)
    total_score = total_score * penalty_factor
```

**影响说明**：
- 一致性评分 ≥ 0.7：无惩罚
- 一致性评分 < 0.7：总分乘以评分值
- 一致性评分过低：触发重新生成

## 六、API接口参考

### 6.1 核心接口

#### analyze_chapter_vs_settings

分析单章节与设定的冲突。

**参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| chapter_text | str | 章节文本内容 |
| current_settings | dict | 当前项目设定 |
| chapter_id | str | 章节ID或标题 |

**返回**：`ConsistencyReport`

**示例**：
```python
report = plugin.analyze_chapter_vs_settings(
    chapter_text="林逸拔出长剑，施展御剑术...",
    current_settings={
        "outline": "第一章：主角觉醒...",
        "characters": [{"name": "林逸", "ability": "基础剑法"}],
        "worldview": "修仙世界..."
    },
    chapter_id="chap_003"
)
```

#### analyze_project

分析整个项目的一致性。

**参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| project_data | dict | 完整项目数据 |
| options | dict | 分析选项 |

**返回**：`ConsistencyReport`

#### generate_corrections

根据冲突报告生成修正后的设定。

**参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| report | ConsistencyReport | 一致性分析报告 |
| current_settings | dict | 当前项目设定 |
| options | dict | 修正选项 |

**返回**：
```python
{
    "updated_outline": str,
    "updated_characters": List[Dict],
    "updated_worldview": str,
    "suggestions": List[str],
    "backup": Optional[Dict]
}
```

### 6.2 数据模型

#### ConsistencyIssue（冲突项）

| 字段 | 类型 | 说明 |
|------|------|------|
| issue_id | str | 冲突项唯一标识 |
| issue_type | ConsistencyIssueType | 冲突类型 |
| severity | ConsistencySeverity | 严重程度 |
| description | str | 冲突描述 |
| suggested_fix | str | 建议修正方案 |
| original_content | str | 原始设定内容 |
| chapter_reference | str | 引发冲突的章节ID |
| element_name | str | 冲突涉及的元素名称 |
| confidence | float | 检测置信度（0-1） |

#### ConsistencyReport（分析报告）

| 字段 | 类型 | 说明 |
|------|------|------|
| report_id | str | 报告唯一标识 |
| project_name | str | 项目名称 |
| chapters_analyzed | int | 分析的章节数量 |
| issues | List[ConsistencyIssue] | 冲突项列表 |
| summary | str | 分析摘要 |
| high_priority_count | int | 高优先级冲突数量 |
| medium_priority_count | int | 中等优先级冲突数量 |
| low_priority_count | int | 低优先级冲突数量 |

## 七、配置说明

### 7.1 config.yaml 配置

```yaml
# LLM配置（必需）
provider: DeepSeek  # 或 OpenAI
api_key: your-api-key
local_url: http://localhost:11434/v1  # 本地模型URL（可选）
model: deepseek-chat
temperature: 0.7

# 分析配置（可选）
cache_ttl: 3600           # 缓存过期时间（秒）
max_analysis_retries: 3   # 分析失败重试次数
analysis_timeout: 60      # 分析超时时间（秒）
```

## 八、最佳实践

### 8.1 分析时机

| 时机 | 建议 |
|------|------|
| 单章节生成后 | 运行快速检查 |
| 完成一卷内容 | 运行完整分析 |
| 项目定稿前 | 运行全面检查 |
| 设定修改后 | 重新分析相关章节 |

### 8.2 修正策略

| 冲突级别 | 处理建议 |
|----------|----------|
| 🔴 高优先级 | 必须修正，否则影响整体逻辑 |
| 🟡 中优先级 | 建议修正，影响阅读体验 |
| 🟢 低优先级 | 可选修正，细节问题 |

### 8.3 工作流建议

```
1. 生成章节
   ↓
2. 自动逆向反馈检查（集成在质量验证器中）
   ↓
3. 查看冲突报告
   ↓
4. 决定处理方式：
   - 应用修正（更新设定）
   - 修改章节内容
   - 忽略低优先级冲突
   ↓
5. 继续下一章节生成
```

## 九、常见问题

### Q1: 分析结果不准确怎么办？

**可能原因**：
1. 设定信息不完整
2. LLM理解偏差
3. 章节内容截断

**解决方案**：
- 补充完整的项目设定
- 使用更强大的模型（如 GPT-4）
- 检查章节是否被截断

### Q2: 分析时间过长怎么办？

**可能原因**：
1. 首次分析需调用LLM
2. 章节数量过多

**解决方案**：
- 使用缓存机制（自动启用）
- 分批分析章节
- 配置本地模型加速

### Q3: 如何禁用自动分析？

修改配置文件：
```yaml
reverse_feedback:
  auto_analyze: false
```

### Q4: 修正后的设定可以回滚吗？

可以。应用修正时会自动备份原始设定：
```python
corrections = plugin.generate_corrections(
    report=report,
    current_settings=settings,
    options={"preserve_original": True}
)
backup = corrections["backup"]  # 原始设定备份
```
