### 第三章：微内核核心引擎

**评审结论**：⚠️ 通过（含 2 项高风险问题）

#### 🔴 P0-001：EventBus.publish() 异步语义误导

**位置**：3.1.2 `EventBus.publish()` 方法注释

**问题**：方法注释标注为"异步"，但实现是**同步遍历** handler（无 Thread / ThreadPool / asyncio），仅"锁外调用"而非真正异步。这会造成两个后果：
- 调用者误以为 publish 立即返回，实际上 handler 执行时间会阻塞调用线程
- 若某个 handler 耗时（如 LLM 调用），会导致 UI 线程卡死（若从 UI 线程调用）

**修正建议**：
```python
# 方案A（推荐）：使用 ThreadPoolExecutor 实现真异步
from concurrent.futures import ThreadPoolExecutor

class EventBus:
    def __init__(self):
        self._executor = ThreadPoolExecutor(max_workers=10)
    
    def publish(self, event_type, data, source=None):
        """发布事件（真正异步 - 提交到线程池）"""
        event = Event(type=event_type, data=data, source=source)
        with self._lock:
            handlers = list(self._handlers.get(event_type, []))
        
        for handler_info in handlers:
            self._executor.submit(self._safe_invoke, handler_info["handler"], event)
    
    def _safe_invoke(self, handler, event):
        try:
            handler(event)
        except Exception as e:
            # 记录到 dead letter queue
            pass

# 方案B（简单）：改注释为"同步调用"，从 UI 调用时需要先 after(0, ...)
```

#### 🔴 P0-002：PluginRegistry 状态转换不完整

**位置**：3.2.2 `PluginRegistry.activate()` / `deactivate()`

**问题**：当前状态机仅有 LOADED ↔ ACTIVE 两条路径，缺少：
1. ERROR 状态的恢复路径（`ERROR → LOADED`，如重试加载后）
2. UNLOADING 状态的完成路径（`UNLOADING → (removed)`）
3. activate/deactivate 失败时是否转为 ERROR 状态

**修正建议**：补充完整状态机文档：
```
LOADED → ACTIVE        (activate 成功)
ACTIVE → LOADED        (deactivate 成功)
LOADED → ERROR         (initialize 抛出异常)
ACTIVE → ERROR         (execute 连续失败超阈值)
ERROR  → LOADED        (重试初始化成功)
LOADED → UNLOADING     (卸载请求)
ACTIVE → UNLOADING     (卸载请求，需先 deactivate)
UNLOADING → (removed)  (shutdown + cleanup 完成)
```

**建议在代码中**通过 `_validate_transition(from_state, to_state)` 强制校验非法转换。

#### 🟡 P1-001：ServiceLocator 循环依赖检测有 Bug

**位置**：3.3.2 `ServiceLocator.check_circular_dependency()`

**问题**：当前实现依赖 `service.dependencies` 属性存在（`hasattr(instance, 'dependencies')`），但 `BasePlugin` 的 `dependencies` 存储在 `metadata.dependencies`（字符串列表）而非直接属性。`ServiceLocator` 管理的是 **服务对象**（不一定是插件），无法统一用 `metadata.dependencies` 访问。

**建议**：明确 ServiceLocator 管理的服务是否需要循环依赖检测，若需要，定义统一的 `IDependencyAware` 协议：
```python
class IDependencyAware(Protocol):
    @property
    def dependencies(self) -> List[Type]: ...
```

#### 🟡 P1-002：ConfigManager 并发写入未保护

**位置**：3.4.3 `ConfigManager.set_config()`

**问题**：`_set_nested()` 和 `_save_config()` 之间没有锁，多线程同时 `set_config()` 可能导致配置文件写入竞争或数据丢失。

**修正建议**：
```python
def set_config(self, key_path: str, value: Any):
    with self._write_lock:  # 新增写锁
        old_value = self.get_config(key_path)
        self._set_nested(key_path, value)
        self._save_config()
    
    if self._event_bus:
        self._event_bus.publish("config.changed", {...})
```

### 第四章：插件系统

**评审结论**：✅ 通过（含 1 项高风险、2 项中风险问题）

#### 🔴 P0-003：BasePlugin.execute() 签名过于宽泛

**位置**：4.2.1 `BasePlugin.execute(*args, **kwargs)`

**问题**：`execute()` 的签名是 `(*args, **kwargs) -> Any`，完全无类型约束，与下方具体接口（`AnalyzerPlugin.analyze(content, options)`、`GeneratorPlugin.generate(request)`、`ValidatorPlugin.validate(content, context)`）形成不一致。调用方无法依赖 `execute()` 进行统一调用，导致这个方法实际上毫无价值，还会引导开发者错误地通过 `plugin.execute()` 调用功能。

**修正建议**：
- 将 `BasePlugin.execute()` 改为**可选（非抽象）**的向后兼容方法，标注 `@deprecated`
- 强制要求所有具体功能通过专用接口方法调用（`analyze() / generate() / validate()`）
- 或者删除 `execute()` 抽象方法，各子接口定义自己的核心方法

#### 🟡 P1-003：DependencyResolver 拓扑排序逻辑有缺陷

**位置**：4.3.2 `DependencyResolver.resolve()`

**问题**：`in_degree` 的计算逻辑有误——当前代码将 **被依赖插件** 的入度+1（`in_degree[dep_id] += 1`），但 Kahn 算法应该是**依赖方**的入度计算（即 plugin A 依赖 plugin B，则 A 的入度+1，而非 B）。现有代码会导致加载顺序反转——被依赖的插件可能最后加载。

**修正建议**：
```python
# 正确的入度计算：依赖方的入度 = 其依赖的插件数
for plugin_id, metadata in plugins.items():
    in_degree[plugin_id] = len(metadata.dependencies)
    for dep_id in metadata.dependencies:
        if dep_id not in graph:
            errors.append(...)

# Kahn 算法：入度为0的节点（无未满足依赖）先加载
queue = [pid for pid, degree in in_degree.items() if degree == 0]

# 遍历时，当 dep 被加载后，依赖它的插件入度减1
# 注意：需要反向图（谁依赖了当前节点）
```
