## 一、数据库表结构设计评审

### 1.1 现有表结构分析

架构文档定义了4个核心表：

```sql
-- schema_version表
CREATE TABLE IF NOT EXISTS schema_version (
    version INTEGER PRIMARY KEY,
    applied_at TEXT DEFAULT (datetime('now', 'localtime'))
);

-- generation_history表
CREATE TABLE IF NOT EXISTS generation_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    chapter_title TEXT NOT NULL,
    iteration_count INTEGER DEFAULT 1,
    final_score REAL,
    word_count INTEGER,
    created_at TEXT DEFAULT (datetime('now', 'localtime')),
    metadata TEXT
);

-- plugin_state表
CREATE TABLE IF NOT EXISTS plugin_state (
    plugin_id TEXT PRIMARY KEY,
    state TEXT NOT NULL,
    enabled INTEGER DEFAULT 1,
    load_count INTEGER DEFAULT 0,
    last_loaded TEXT,
    config TEXT,
    error_count INTEGER DEFAULT 0,
    last_error TEXT
);

-- operation_log表
CREATE TABLE IF NOT EXISTS operation_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    operation TEXT NOT NULL,
    module TEXT,
    status TEXT,
    duration_ms INTEGER,
```

### 1.2 设计优势

1. **版本管理机制**：`schema_version`表支持数据库迁移，符合最佳实践
2. **WAL模式**：启用读写并发，性能优于默认模式
3. **基础完整性**：主键定义合理，默认值设置合理

### 1.3 设计缺陷与建议（按优先级）

#### P0级别问题（严重）

**P0-DB-001：所有表缺少索引，性能将严重劣化**

问题：
- 4个表仅定义主键，无任何辅助索引
- `generation_history`表频繁查询`chapter_title`、`created_at`，但无索引
- `operation_log`表根据`module`、`status`、`created_at`筛选，但无索引
- `plugin_state`表根据`enabled`状态查询活跃插件，但无索引

影响：
- 随着数据增长（生成历史、日志累计），查询性能呈O(n)下降
- 典型场景：查询最近1周的生成历史、查询某模块的所有错误日志，全表扫描

建议索引方案：

```sql
-- generation_history表索引
CREATE INDEX idx_generation_created ON generation_history(created_at DESC);
CREATE INDEX idx_generation_title ON generation_history(chapter_title);
CREATE INDEX idx_generation_score ON generation_history(final_score);

-- plugin_state表索引
CREATE INDEX idx_plugin_enabled ON plugin_state(enabled);

-- operation_log表索引
CREATE INDEX idx_oplog_module ON operation_log(module);
CREATE INDEX idx_oplog_status ON operation_log(status);
CREATE INDEX idx_oplog_created ON operation_log(created_at DESC);
CREATE INDEX idx_oplog_module_status ON operation_log(module, status);

-- 复合索引优化（高频查询路径）
CREATE INDEX idx_oplog_time_module ON operation_log(created_at DESC, module);
```

优先级：立即执行，在迁移V1中包含索引创建。

**P0-DB-002：缺少外键约束，数据一致性无保障**

问题：
- `generation_history.metadata`字段可能包含插件ID引用，但无外键约束
- `plugin_state.state`字段值域未定义（如'loaded'、'unloaded'等），但无CHECK约束或枚举表

影响：
- 删除插件后，`generation_history`中可能残留引用该插件的历史记录
- `plugin_state.state`字段可能插入非法值（如拼写错误），导致状态机失效

建议方案：

```sql
-- 方案A：添加外键约束（推荐）
-- 如果metadata是JSON包含plugin_id，建议单独建关联表
CREATE TABLE generation_plugin_mapping (
    generation_id INTEGER NOT NULL,
    plugin_id TEXT NOT NULL,
    role TEXT NOT NULL,  -- 'analyzer', 'generator', 'validator'
    FOREIGN KEY (generation_id) REFERENCES generation_history(id) ON DELETE CASCADE,
    FOREIGN KEY (plugin_id) REFERENCES plugin_state(plugin_id) ON DELETE RESTRICT,
    PRIMARY KEY (generation_id, plugin_id, role)
);

-- 方案B：添加CHECK约束（快速修复）
ALTER TABLE plugin_state ADD CONSTRAINT chk_state_valid
    CHECK (state IN ('UNLOADED', 'LOADING', 'LOADED', 'UNLOADING', 'ERROR'));
```

优先级：MVP阶段建议执行方案B快速修复，V2迭代时重构为方案A。

**P0-DB-003：operation_log表未分区，日志清理将极其低效**

问题：
- `operation_log`表用于记录所有操作日志，数据增长速度快
- 架构文档未定义日志归档/清理策略
- 日志表无分区，直接DELETE清理导致大量碎片和性能下降

影响：
- 数月后日志表可能达数百万行，查询和写入性能急剧下降
- 清理历史日志需全表扫描或依赖索引，耗时不可预测

建议方案：

```sql
-- 方案A：按时间分区（SQLite 3.25+支持）
-- 但SQLite分区功能有限，建议用表继承模式
CREATE TABLE operation_log_2026_q1 (
    CHECK (created_at >= '2026-01-01' AND created_at < '2026-04-01')
) INHERITS (operation_log);

-- 方案B：定期归档（推荐，SQLite更友好）
-- 创建归档表和清理脚本
CREATE TABLE operation_log_archive (
    LIKE operation_log INCLUDING DEFAULTS
);

-- 清理脚本（保留最近3个月，超过的归档）
INSERT INTO operation_log_archive
SELECT * FROM operation_log
WHERE created_at < datetime('now', '-3 months');
DELETE FROM operation_log WHERE created_at < datetime('now', '-3 months');
```

优先级：立即在迁移V1中添加`operation_log_archive`表，V2实现自动化归档脚本。

## 二、并发优化设计评审

### 2.1 SQLite并发设计

**优点**：
- WAL模式设计正确，支持读写并发
- 连接池上限10个合理（SQLite默认限制，避免资源耗尽）

**问题**：

**P1-DB-004：连接池未在数据库层实现，仅架构描述**

问题：
- 架构文档8.3节提及"连接池（最多10个连接）"，但未提供实现代码
- 数据库迁移模块`DatabaseMigration`无连接池相关代码

影响：
- 多线程环境下，每个线程可能创建独立连接，超出SQLite限制
- 无连接复用，频繁创建/销毁连接开销大

建议实现：

```python
# infrastructure/database_pool.py
import sqlite3
import threading
from queue import Queue
from typing import Optional

class SQLiteConnectionPool:
    def __init__(self, db_path: str, max_connections: int = 10):
        self.db_path = db_path
        self.max_connections = max_connections
        self._pool: Queue[sqlite3.Connection] = Queue(max_connections)
        self._lock = threading.Lock()
        self._active_connections = 0

        for _ in range(max_connections):
            conn = self._create_connection()
            self._pool.put(conn)

    def _create_connection(self) -> sqlite3.Connection:
        conn = sqlite3.connect(self.db_path, check_same_thread=False)
        conn.execute('PRAGMA journal_mode=WAL')  # 确保WAL模式
        conn.execute('PRAGMA synchronous=NORMAL')  # 平衡性能与安全
        conn.execute('PRAGMA foreign_keys=ON')  # 启用外键约束
        return conn

    def get_connection(self) -> sqlite3.Connection:
        with self._lock:
            if self._active_connections >= self.max_connections:
                raise Exception("Connection pool exhausted")
            self._active_connections += 1
        return self._pool.get()

    def release_connection(self, conn: sqlite3.Connection):
        try:
            self._pool.put_nowait(conn)
        except:
            conn.close()
        finally:
            with self._lock:
                self._active_connections -= 1
```

优先级：立即实现并集成到`DatabaseMigration`和所有数据库操作模块。

**P1-DB-005：事务超时30秒未在代码中实现**

问题：
- 架构文档8.3节提及"事务超时（30秒）"，但无实现代码
- SQLite不支持事务超时参数，需应用层实现

建议实现：

```python
import signal
from contextlib import contextmanager

class TransactionTimeoutError(Exception):
    pass

@contextmanager
def transaction_with_timeout(conn: sqlite3.Connection, timeout_seconds: int = 30):
    """带超时控制的事务上下文管理器"""
    def timeout_handler(signum, frame):
        raise TransactionTimeoutError(f"Transaction exceeded {timeout_seconds}s limit")

    signal.signal(signal.SIGALRM, timeout_handler)
    signal.alarm(timeout_seconds)

    try:
        conn.execute('BEGIN TRANSACTION')
        yield conn
        conn.commit()
    except Exception as e:
        conn.rollback()
        raise
    finally:
        signal.alarm(0)  # 取消定时器
```

注意：Windows下`signal.SIGALRM`不可用，需用`threading.Timer`替代。

优先级：V2实现，需跨平台兼容性处理。

### 2.2 事件总线并发设计

**优点**：
- 线程池大小（10个工作线程）合理
- 超时控制（5秒）设计合理

**问题（已在评审结果01中指出）**：

**P0-001：EventBus.publish()注释"异步"但实现同步**

问题：
- `EventBus.publish()`方法遍历所有handler同步调用
- 无ThreadPoolExecutor/asyncio支持，仅"锁外调用"非真正异步
- UI线程调用会卡死（已在评审结果01详细说明）

建议（复用评审结果01方案A）：

```python
from concurrent.futures import ThreadPoolExecutor

class EventBus:
    def __init__(self):
        self._executor = ThreadPoolExecutor(max_workers=10)  # 线程池

    def publish(self, event_type: str, data: Any):
        """真异步发布"""
        handlers = self._handlers.get(event_type, [])
        futures = []

        for h in handlers:
            future = self._executor.submit(h['handler'], event_data)
            futures.append(future)

        # 等待所有handler完成或超时
        from concurrent.futures import wait, FIRST_EXCEPTION, TimeoutError
        try:
            wait(futures, timeout=5.0, return_when=FIRST_EXCEPTION)
        except TimeoutError:
            # 超时处理：记录日志，不阻塞
            pass
```

优先级：立即执行（已在评审结果01列为P0）。

```python
# infrastructure/cache_manager.py
from typing import Dict, Any, Optional
from datetime import datetime, timedelta

class CacheManager:
    def __init__(self):
        self._prompt_cache: Dict[str, Any] = {}  # 永久
        self._response_cache: Dict[str, tuple] = {}  # (value, expiry_time)
        self._config_cache: Dict[str, tuple] = {}  # (value, expiry_time)
        self._context_cache: Dict[str, Any] = {}  # 上下文缓存

    def set_config(self, plugin_id: str, config: Any, ttl: int = 3600):
        """配置缓存（默认1小时TTL）"""
        expiry = datetime.now() + timedelta(seconds=ttl)
        self._config_cache[plugin_id] = (config, expiry)

    def get_config(self, plugin_id: str) -> Optional[Any]:
        if plugin_id in self._config_cache:
            config, expiry = self._config_cache[plugin_id]
            if datetime.now() < expiry:
                return config
            del self._config_cache[plugin_id]
        return None

    def invalidate_config(self, plugin_id: str):
        """插件配置变更时失效"""
        if plugin_id in self._config_cache:
            del self._config_cache[plugin_id]
```

### 4.2 设计问题

**P1-DB-007：迁移机制不完整，缺少关键实现**

问题：
1. `get_current_version()`未实现
2. `migrate()`方法无错误处理和回滚机制
3. 迁移前无备份机制
4. 迁移后无版本记录更新

建议完整实现：

```python
import shutil
from pathlib import Path

class DatabaseMigration:
    def __init__(self, db_path: str):
        self.db_path = db_path
        self._backup_dir = Path(db_path).parent / "db_backups"
        self._backup_dir.mkdir(exist_ok=True)
        self._migrations: Dict[int, Callable] = {
            1: self._migration_v1,
            2: self._migration_v2,  # 示例：添加索引
        }

    def get_current_version(self) -> int:
        """获取当前数据库版本"""
        try:
            with sqlite3.connect(self.db_path) as conn:
                cursor = conn.execute(
                    "SELECT version FROM schema_version ORDER BY version DESC LIMIT 1"
                )
                row = cursor.fetchone()
                return row[0] if row else 0
        except sqlite3.OperationalError:
            return 0  # schema_version表不存在，视为版本0

    def backup_database(self) -> str:
        """迁移前备份"""
        timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
        backup_path = self._backup_dir / f"db_backup_{timestamp}.db"
        shutil.copy2(self.db_path, backup_path)
        return str(backup_path)

    def migrate(self) -> bool:
        """执行迁移（带备份和回滚）"""
        try:
            current_version = self.get_current_version()
            if current_version >= self.CURRENT_VERSION:
                return True

            # 迁移前备份
            backup_path = self.backup_database()

            for version in range(current_version + 1, self.CURRENT_VERSION + 1):
                migration = self._migrations.get(version)
                if not migration:
                    raise ValueError(f"Migration version {version} not defined")

                print(f"Executing migration v{version}...")
                migration()

                # 更新版本记录
                with sqlite3.connect(self.db_path) as conn:
                    conn.execute(
                        "INSERT INTO schema_version (version) VALUES (?)",
                        (version,)
                    )
                    conn.commit()

            print(f"Migration completed: v{current_version} → v{self.CURRENT_VERSION}")
            print(f"Backup saved to: {backup_path}")
            return True

        except Exception as e:
            print(f"Migration failed: {e}")
            # 回滚：恢复备份
            if 'backup_path' in locals():
                shutil.copy2(backup_path, self.db_path)
                print(f"Rollback completed using backup: {backup_path}")
            return False
```

优先级：立即实现，V1迁移必须包含完整错误处理。

## 六、结论

《项目总体架构设计说明书V1.0》的数据库设计**基础正确**（WAL模式、版本管理、缓存概念），但**实现细节不完整**（缺少索引、外键、连接池、错误处理）。

**建议**：按上述路线图分三阶段修复，第一阶段必须在MVP前完成（索引、连接池、迁移机制、EventBus异步）。
