### 第三章：微内核核心引擎

**评审结论**：⚠️ 通过（含2项高风险问题）

#### 🔴 P0-001：EventBus.publish()注释与实现不符

**位置**：3.1.2 `EventBus.publish()` 方法注释

**问题**：注释标注"异步"，但实现是同步遍历handler（无ThreadPoolExecutor/asyncio），仅锁外调用而非真正异步。这会导致：
- 调用者误以为publish立即返回，实际handler执行时间会阻塞调用线程
- 若handler耗时（如LLM调用），UI线程调用时会卡死

**修正建议**：
```python
# 方案A（推荐）：使用ThreadPoolExecutor实现真异步
from concurrent.futures import ThreadPoolExecutor

class EventBus:
    def __init__(self):
        self._executor = ThreadPoolExecutor(max_workers=10)
        self._dead_letter_queue: List[Dict] = []
    
    def publish(self, event_type, data, source=None):
        """发布事件（真正异步 - 提交到线程池）"""
        event = Event(type=event_type, data=data, source=source)
        with self._lock:
            handlers = list(self._handlers.get(event_type, []))
        
        for handler_info in handlers:
            self._executor.submit(self._safe_invoke, handler_info, event)
    
    def _safe_invoke(self, handler_info, event):
        try:
            handler_info["handler"](event)
        except Exception as e:
            # 失败重试逻辑或进入死信队列
            pass
```

#### 🔴 P0-002：PluginRegistry状态机不完整

**位置**：3.2.2 PluginRegistry

**问题**：当前仅有LOADED↔ACTIVE两条路径，缺少：
1. ERROR状态的恢复路径
2. UNLOADING状态的完成路径
3. 激活/停用失败时是否转为ERROR

**修正建议**：补充完整状态机并添加_validate_transition()校验
```
LOADED → ACTIVE        (activate成功)
ACTIVE → LOADED        (deactivate成功)
LOADED → ERROR         (initialize异常)
ACTIVE → ERROR         (execute连续失败)
ERROR → LOADED         (重试成功)
LOADED → UNLOADING     (卸载请求)
ACTIVE → UNLOADING     (需先deactivate)
UNLOADING → (removed)  (shutdown+cleanup完成)
```

#### 🟡 P1-001：ServiceLocator循环依赖检测路径不明确

**位置**：3.3.2 check_circular_dependency()

**问题**：依赖hasattr(instance, 'dependencies')，但ServiceLocator管理的是服务对象（不一定是插件），dependencies存储路径不统一（BasePlugin存于metadata）。

**建议**：明确ServiceLocator检测范围，或定义IDependencyAware协议统一接口。

#### 🟡 P1-002：ConfigManager并发写入未保护

**位置**：3.4.3 set_config()

**问题**：_set_nested()和_save_config()之间无锁，多线程同时set_config()可能数据丢失。

**修正建议**：添加写锁：
```python
def set_config(self, key_path: str, value: Any):
    with self._write_lock:
        old_value = self.get_config(key_path)
        self._set_nested(key_path, value)
        self._save_config()
```

### 第四章：插件系统

**评审结论**：⚠️ 通过（含1项高风险问题）

#### 🔴 P0-003：BasePlugin.execute()签名无实际价值

**位置**：4.2.1 BasePlugin.execute(*args, **kwargs)

**问题**：宽泛签名与专用接口方法（analyze()/generate()/validate()）冲突，且无类型约束，调用方无法统一调用，实际价值为零。

**修正建议**：
- 将execute()降级为可选的向后兼容方法，添加@deprecated标注
- 强制通过专用接口方法调用功能
- 或直接删除BasePlugin.execute()抽象方法

#### 🟡 P1-003：DependencyResolver入度计算方向

**位置**：4.3.2 DependencyResolver.resolve()

**问题**：代码将"被依赖插件"的入度+1，但Kahn算法中"依赖方"的入度应等于其依赖数。建议复核算法正确性。

**修正建议**：
```python
# 正确：依赖方的入度 = 其未满足的依赖数
for plugin_id, metadata in plugins.items():
    in_degree[plugin_id] = len(metadata.dependencies)
# 无依赖的插件入度为0，优先加载
queue = [pid for pid, d in in_degree.items() if d == 0]
```

#### 🟡 P1-004：plugin.json格式未定义

**位置**：4.4 插件目录结构

**问题**：多处引用plugin.json但未给出JSON格式示例，开发者无法编写插件。

**建议补充**：
```json
{
  "id": "outline-parser-v3",
  "name": "大纲解析器",
  "version": "3.0.0",
  "plugin_type": "analyzer",
  "api_version": "1.0",
  "priority": 100,
  "dependencies": [],
  "entry_class": "OutlineParserPlugin"
}
```
