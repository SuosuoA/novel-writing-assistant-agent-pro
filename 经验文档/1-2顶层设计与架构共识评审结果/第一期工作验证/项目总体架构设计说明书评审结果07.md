### 2.2 问题识别

**P1-DM-001：缺少ORM映射层，数据库操作需手写SQL**

- **影响**: 开发效率低、类型安全无法保障、查询逻辑分散
- **建议**: 引入SQLModel（SQLAlchemy 2.0 + Pydantic v2）作为ORM层

```python
# 建议方案
from sqlmodel import SQLModel, Field as SQLField
from pydantic import BaseModel

class GenerationHistory(SQLModel, table=True):
    """数据库表模型"""
    id: Optional[int] = SQLField(default=None, primary_key=True)
    chapter_title: str = SQLField(index=True)

class GenerationRequest(BaseModel):
    """Pydantic验证模型（保持现有设计）"""
    pass
```

**P2-DM-002：数据模型缺少版本兼容性设计**

- **影响**: 未来模型变更可能导致数据不兼容
- **建议**: 添加`schema_version`字段到核心模型

### 3.2 P0级别问题（必须立即修复）

#### **P0-DB-001：所有表缺少索引，查询性能将严重劣化**

**问题描述**：
- `generation_history`表无索引，按时间/标题查询将全表扫描
- `operation_log`表无索引，日志查询和清理性能极差
- `plugin_state`表无索引，状态查询效率低

**影响评估**：
- 数据量达10万行时，查询延迟可能超过1秒
- 日志表随时间增长，清理操作将锁定整个表

**改进建议**：

```sql
-- generation_history表索引
CREATE INDEX idx_generation_created ON generation_history(created_at DESC);
CREATE INDEX idx_generation_title ON generation_history(chapter_title);
CREATE INDEX idx_generation_score ON generation_history(final_score);

-- plugin_state表索引
CREATE INDEX idx_plugin_enabled ON plugin_state(enabled);

-- operation_log表索引（复合索引优化高频查询）
CREATE INDEX idx_oplog_module ON operation_log(module);
CREATE INDEX idx_oplog_status ON operation_log(status);
CREATE INDEX idx_oplog_created ON operation_log(created_at DESC);
CREATE INDEX idx_oplog_module_status ON operation_log(module, status);
CREATE INDEX idx_oplog_time_module ON operation_log(created_at DESC, module);
```

#### **P0-DB-002：缺少外键约束，数据一致性无保障**

**问题描述**：
- `generation_history.metadata`可能包含插件ID引用，但无外键约束
- `plugin_state.state`字段值域未定义，无CHECK约束

**影响评估**：
- 删除插件后，历史记录中可能残留无效引用
- 非法状态值（如拼写错误）可能导致状态机失效

**改进建议**：

```sql
-- 方案A：添加外键关联表（推荐）
CREATE TABLE generation_plugin_mapping (
    generation_id INTEGER NOT NULL,
    plugin_id TEXT NOT NULL,
    role TEXT NOT NULL,  -- 'analyzer', 'generator', 'validator'
    FOREIGN KEY (generation_id) REFERENCES generation_history(id) ON DELETE CASCADE,
    FOREIGN KEY (plugin_id) REFERENCES plugin_state(plugin_id) ON DELETE RESTRICT,
    PRIMARY KEY (generation_id, plugin_id, role)
);

-- 方案B：添加CHECK约束（快速修复）
ALTER TABLE plugin_state ADD CONSTRAINT chk_state_valid
    CHECK (state IN ('UNLOADED', 'LOADING', 'LOADED', 'UNLOADING', 'ERROR'));
```

#### **P0-DB-003：operation_log表未分区，日志清理将极其低效**

**问题描述**：
- 操作日志数据增长速度快，数月后可能达数百万行
- 无分区设计，DELETE清理导致大量碎片和性能下降

**改进建议**：

```sql
-- 创建归档表
CREATE TABLE operation_log_archive (
    LIKE operation_log INCLUDING DEFAULTS
);

-- 清理脚本（保留最近3个月）
INSERT INTO operation_log_archive
SELECT * FROM operation_log
WHERE created_at < datetime('now', '-3 months');
DELETE FROM operation_log WHERE created_at < datetime('now', '-3 months');
```

### 3.3 P1级别问题（重要）

#### **P1-DB-004：连接池未实现，仅架构描述**

**问题描述**：
- 架构文档提及"连接池（最多10个连接）"，但未提供实现代码
- `DatabaseMigration`类无连接池相关代码

**影响评估**：
- 多线程环境下，每个线程可能创建独立连接，超出SQLite限制
- 无连接复用，频繁创建/销毁连接开销大

**改进建议**：

```python
class SQLiteConnectionPool:
    """SQLite连接池实现"""
    
    def __init__(self, db_path: str, max_connections: int = 10):
        self.db_path = db_path
        self.max_connections = max_connections
        self._pool = queue.Queue(maxsize=max_connections)
        self._lock = threading.RLock()
        self._active_connections = 0
        
        # 预创建连接
        for _ in range(max_connections):
            conn = self._create_connection()
            self._pool.put(conn)
    
    def _create_connection(self) -> sqlite3.Connection:
        conn = sqlite3.connect(self.db_path, check_same_thread=False)
        conn.execute('PRAGMA journal_mode=WAL')
        conn.execute('PRAGMA synchronous=NORMAL')
        conn.execute('PRAGMA foreign_keys=ON')
        # 性能优化PRAGMA
        conn.execute('PRAGMA cache_size=-64000')  # 64MB缓存
        conn.execute('PRAGMA temp_store=MEMORY')
        conn.execute('PRAGMA mmap_size=268435456')  # 256MB内存映射
        return conn
    
    def get_connection(self) -> sqlite3.Connection:
        with self._lock:
            if self._active_connections >= self.max_connections:
                raise Exception("Connection pool exhausted")
            self._active_connections += 1
        return self._pool.get()
    
    def release_connection(self, conn: sqlite3.Connection):
        self._pool.put(conn)
        with self._lock:
            self._active_connections -= 1
```

#### **P1-DB-005：事务超时30秒未在代码中实现**

**问题描述**：
- 架构文档提及"事务超时（30秒）"，但无实际实现

**改进建议**：

```python
def execute_with_timeout(conn: sqlite3.Connection, query: str, 
                         params: tuple = (), timeout: float = 30.0):
    """带超时的事务执行"""
    import signal
    
    def timeout_handler(signum, frame):
        raise TimeoutError(f"Query timeout after {timeout}s")
    
    signal.signal(signal.SIGALRM, timeout_handler)
    signal.setitimer(signal.ITIMER_REAL, timeout)
    
    try:
        cursor = conn.execute(query, params)
        conn.commit()
        return cursor
    finally:
        signal.alarm(0)
```

### 4.2 P0级别问题

#### **P0-DF-001：EventBus.publish()伪异步问题**

**问题描述**：
- 注释标注"异步"，但实现是同步遍历handler
- 无ThreadPoolExecutor/asyncio支持，仅"锁外调用"非真正异步
- UI线程调用会卡死

**影响评估**：
- 若handler耗时（如LLM调用），UI线程调用时会卡死
- 调用者误以为publish立即返回

**改进建议**：

```python
from concurrent.futures import ThreadPoolExecutor

class EventBus:
    def __init__(self):
        self._executor = ThreadPoolExecutor(max_workers=10)
        self._dead_letter_queue: List[Dict] = []
    
    def publish(self, event_type, data, source=None):
        """真正异步发布"""
        event = Event(type=event_type, data=data, source=source)
        with self._lock:
            handlers = list(self._handlers.get(event_type, []))
        
        for handler_info in handlers:
            future = self._executor.submit(
                self._execute_handler, 
                handler_info, 
                event
            )
            future.add_done_callback(
                lambda f: self._handle_completion(f, event, handler_info)
            )
```

### 4.3 缓存一致性风险

**缓存策略定义**（来自架构文档）：

| 缓存类型 | 缓存内容 | 过期策略 | 容量 |
|----------|----------|----------|------|
| 插件缓存 | 已加载插件实例 | 永久 | 全部 |
| 风格缓存 | 风格特征向量 | 24小时 | 100MB |
| 提示词缓存 | 构建的提示词 | 永久 | 1000条 |
| 响应缓存 | LLM响应结果 | 1小时 | 500MB |

**风险识别**：

**P1-DF-002：缓存与数据库数据一致性无保障**

- 插件状态变更时，缓存未定义失效策略
- 建议添加缓存失效事件机制

```python
class CacheManager:
    def invalidate_plugin_cache(self, plugin_id: str):
        """插件配置变更时失效"""
        if plugin_id in self._config_cache:
            del self._config_cache[plugin_id]
            # 发布缓存失效事件
            self._event_bus.publish("cache.invalidated", {
                "type": "plugin",
                "id": plugin_id
            })
```

class DataNormalizer:
    """数据标准化器"""
    
    @staticmethod
    def normalize_chapter_title(title: str) -> str:
        """标准化章节标题"""
        # 去除首尾空白
        title = title.strip()
        # 统一全角半角符号
        title = unicodedata.normalize('NFKC', title)
        # 限制长度
        return title[:100]
    
    @staticmethod
    def sanitize_metadata(metadata: Dict) -> Dict:
        """清理元数据"""
        # 移除敏感字段
        sensitive_keys = ['api_key', 'password', 'secret']
        return {k: v for k, v in metadata.items() 
                if k not in sensitive_keys}

### 7.2 P0级别安全问题

#### **P0-SEC-001：API密钥明文存储，无加密机制**

**问题描述**：
- `config.yaml`中AI配置包含API密钥，明文存储
- 无密钥管理系统

**影响评估**：
- 配置文件泄露将导致API密钥暴露
- 不符合安全合规要求

**改进建议**：

```python
import keyring
from cryptography.fernet import Fernet

class SecureConfigManager:
    """安全配置管理器"""
    
    def __init__(self):
        self._cipher = self._init_cipher()
    
    def store_api_key(self, provider: str, api_key: str):
        """安全存储API密钥"""
        # 使用系统密钥环
        keyring.set_password("agent-pro", f"{provider}_api_key", api_key)
        
        # 或：使用主密钥加密后存储
        encrypted = self._cipher.encrypt(api_key.encode())
        self._set_config(f"ai.{provider}.api_key_encrypted", 
                        encrypted.decode())
    
    def get_api_key(self, provider: str) -> str:
        """获取API密钥"""
        # 优先从密钥环获取
        key = keyring.get_password("agent-pro", f"{provider}_api_key")
        if key:
            return key
        
        # 从加密配置获取
        encrypted = self._get_config(f"ai.{provider}.api_key_encrypted")
        if encrypted:
            return self._cipher.decrypt(encrypted.encode()).decode()
        
        return None
```

### 7.3 访问控制

**P1-SEC-003：插件权限系统未细化**

- 当前`permissions`字段为字符串列表，无权限验证实现
- 建议实现基于角色的访问控制(RBAC)

```python
class PermissionManager:
    """权限管理器"""
    
    PERMISSIONS = {
        'file.read': '读取文件',
        'file.write': '写入文件',
        'network.access': '网络访问',
        'config.read': '读取配置',
        'config.write': '写入配置',
        'ai.generate': 'AI生成',
    }
    
    def check_permission(self, plugin_id: str, permission: str) -> bool:
        """检查插件权限"""
        plugin = self._registry.get_plugin(plugin_id)
        return permission in plugin.metadata.permissions
```

## 十、高风险数据流程标注

```
🔴 高风险流程：LLM API调用链
UI → EventBus → GeneratorPlugin → LLMClient → 外部API
风险点：
- API密钥明文传输
- 响应数据无缓存一致性保障
- 超时无控制

🔴 高风险流程：插件热插拔
FileSystem → Watchdog → PluginLoader → PluginRegistry → SQLite
风险点：
- 数据库连接未正确释放
- 插件状态变更与缓存不同步

🟡 中风险流程：批量生成
GenerationRequest → 迭代生成 → 验证 → 存储
风险点：
- 大批量数据插入性能
- 事务范围过大
```
