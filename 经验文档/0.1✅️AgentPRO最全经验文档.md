### 2.2 核心设计原则

#### 2.2.1 微内核原则

核心系统仅提供最基本的服务（插件调度、事件分发、依赖管理），所有业务功能通过插件实现。

**核心服务清单**：

| 服务 | 说明 | 关键接口 |
|------|------|---------|
| EventBus | 插件间通信 | `subscribe()`, `publish()`, `publish_sync()` |
| PluginRegistry | 插件管理 | `register()`, `unregister()`, `get_plugin()` |
| ServiceLocator | 依赖注入 | `register_service()`, `get_service()` |
| ConfigManager | 配置管理 | `get_config()`, `set_config()` |

#### 2.2.2 事件驱动原则

插件间通过事件总线通信，解耦依赖。

**事件类型**：

| 事件类型 | 说明 | 典型场景 |
|----------|------|----------|
| 系统事件 | 插件加载/卸载 | `plugin.loaded`, `plugin.unloaded`, `plugin.error` |
| 生成事件 | 生成流程 | `generation.started`, `generation.progress`, `generation.completed` |
| 验证事件 | 质量验证 | `validation.completed` |
| 用户事件 | 用户操作 | `user.generation_requested` |

## 三、插件系统设计

### 3.1 插件类型体系

```
BasePlugin (基础插件)
    ├── ProtocolPlugin (协议插件)
    ├── AIPlugin (AI插件)
    ├── StoragePlugin (存储插件)
    ├── AnalyzerPlugin (分析器插件)
    ├── GeneratorPlugin (生成器插件)
    ├── ValidatorPlugin (验证器插件)
    └── ToolPlugin (工具插件)
```

### 3.2 插件状态管理

**5状态模型**：

```
UNLOADED → LOADED → ACTIVE → UNLOADING → UNLOADED
              ↓         ↓
            ERROR ←←←←←←
```

| 状态 | 说明 | 可转换状态 |
|------|------|-----------|
| `UNLOADED` | 未加载（初始状态） | → LOADED |
| `LOADED` | 已加载，未激活 | → ACTIVE, ERROR, UNLOADING |
| `ACTIVE` | 已激活，可执行 | → LOADED, ERROR, UNLOADING |
| `ERROR` | 错误状态 | → LOADED, UNLOADING |
| `UNLOADING` | 卸载中 | → 从注册表移除 |

### 3.3 热插拔安全权限分级

| 等级 | 类型 | 热重载 | 卸载 | 进程隔离 |
|------|------|--------|------|----------|
| L0 | 官方插件 | ✅ | ✅ | ❌ |
| L1 | 第三方签名 | ✅ | ❌ | ❌ |
| L2 | 未知来源 | ❌ | ❌ | ✅ |
| L3 | 风险插件 | ❌ | ❌ | ❌ |

## 六、AI工程实践经验

### 6.1 Agent系统设计要点

**MasterAgent调度算法**：
- 优先级队列：使用`heapq`实现，时间复杂度O(log n)
- 依赖解析：Kahn算法拓扑排序，检测循环依赖
- 重试机制：指数退避 + jitter，避免thundering herd问题

**Agent执行约束配置**：

| Agent类型 | 超时时间 | 最大重试 | 回退策略 |
|-----------|----------|----------|----------|
| Thinker | 30s | 1次 | 使用缓存结果 |
| Optimizer | 20s | 1次 | 简化优化 |
| Validator | 15s | 0次 | 跳过验证 |
| Planner | 10s | 0次 | 使用默认计划 |

**熔断器状态机**：
```
CLOSED（正常）→ 失败次数≥阈值 → OPEN（熔断）
  ↑                                    ↓
  └──── 半开状态测试成功 ←── HALF_OPEN（半开）
```

### 6.2 API容错与降级方案

**四级容错机制**：
1. **重试层**：指数退避 + jitter，最多3次
2. **熔断层**：失败率>50%且次数≥5时熔断30秒
3. **模型回退**：DeepSeek → OpenAI → Anthropic
4. **缓存降级**：返回最近一次成功结果

**模型回退链**：
```
Primary: DeepSeek-V3
  ↓ (失败)
Fallback1: OpenAI GPT-4o
  ↓ (失败)
Fallback2: Anthropic Claude-3.5-Sonnet
  ↓ (失败)
Cache: 返回最近一次成功结果
```

### 6.3 Prompt模板设计

**5种推理策略**：

| 策略 | 适用场景 | 模板特点 |
|------|----------|----------|
| Zero-shot | 简单任务 | 直接描述任务，无示例 |
| Few-shot | 格式敏感任务 | 提供2-3个输入输出示例 |
| CoT | 逻辑推理任务 | 引导"让我们一步步思考" |
| ToT | 多路径决策 | 生成多个候选路径并评估 |
| ReAct | 工具调用任务 | 交替推理和行动 |

## 七、踩坑与最佳实践

### 7.1 Pydantic v2 迁移踩坑

**问题**：V5.2升级到Pydantic v2时，出现多处兼容性错误

**解决方案**：
1. 全面升级到Pydantic 2.10.6
2. 更新所有模型类的`Config`写法
3. 使用`TypeAdapter`替代重复验证
4. 启用`from_attributes=True`实现零拷贝验证

### 7.2 人设评分低分问题修复（V5.5）

**根因**：性格关键词提取不完整，评分逻辑过于严格

**修复**：
1. 智能提取性格关键词：清理markdown标记
2. 语义扩展匹配：乐观→笑着/开心/笑容等近义词
3. 多维度评分体系：基础分0.5 + 关键词匹配分 + 行为描写分 + 对话描写分
4. 最低分保底0.5分

### 7.3 【本章完】标记缺失修复（V5.6）

**问题**：AI可能未生成标记导致整个生成失败

**修复**：
1. `iterative_generator_v2.py`在最终输出时自动智能补充标记
2. `novel_generator_v3_2.py`做双重保险

### 7.4 线程池关闭参数兼容性

**问题**：Python 3.8不支持`cancel_futures`参数

**解决方案**：
```python
try:
    self._executor.shutdown(wait=True, cancel_futures=True)
except TypeError:
    # Python 3.8及以下版本
    self._executor.shutdown(wait=True)
```

### 7.5 异常处理原则

1. **不吞掉异常**：所有异常必须记录日志
2. **日志完整性**：异常信息 + 相关对象引用 + 关键参数
3. **事件发布**：关键异常发布到EventBus供监控系统感知
4. **资源清理**：使用`atexit`确保退出时清理

### 7.6 资源泄漏防护

**连接池必须实现**：
- 超时回收机制（`max_connection_time`）
- 优雅关闭（等待使用中的资源归还）
- 状态标记（防止关闭后继续使用）

**队列必须有容量限制**：
- `AsyncHandler.DEFAULT_MAX_SIZE = 1000`
- 队列满时拒绝新任务

### 7.7 状态一致性

**关键服务失败处理**：
- 立即终止初始化
- 依赖服务不强制初始化
- 初始化状态必须反映实际结果
