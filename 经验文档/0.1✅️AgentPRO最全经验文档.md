### 2.2 核心设计原则

#### 2.2.1 微内核原则

**定义**：核心系统仅提供最基本的服务（插件调度、事件分发、依赖管理），所有业务功能通过插件实现。

**核心服务清单**：

| 服务 | 说明 | 关键接口 |
|------|------|---------|
| EventBus | 插件间通信 | `subscribe()`, `publish()`, `publish_sync()` |
| PluginRegistry | 插件管理 | `register()`, `unregister()`, `get_plugin()`, `activate()`, `deactivate()` |
| ServiceLocator | 依赖注入 | `register_service()`, `get_service()`, `initialize_all()`, `dispose_all()` |
| ConfigManager | 配置管理 | `get_config()`, `set_config()` |

**非核心服务（插件化）**：
- 大纲解析、风格学习、人物管理、世界观解析
- 上下文构建、迭代生成、加权验证
- 热榜功能、统计分析

#### 2.2.2 榫卯式集成原则

**核心理念**：插件与核心通过标准接口"咬合"，接口定义严格，实现灵活。

**接口设计原则**：
1. **最小化接口**：只定义必要的方法，避免过度设计
2. **向后兼容**：新增方法使用默认实现，不破坏现有插件
3. **显式依赖**：插件依赖在清单中明确声明
4. **清晰边界**：插件不直接访问核心内部状态

#### 2.2.3 事件驱动原则

**核心理念**：插件间通过事件总线通信，解耦依赖。

**事件类型**：

| 事件类型 | 说明 | 典型场景 |
|----------|------|----------|
| 系统事件 | 插件加载/卸载 | `plugin.loaded`, `plugin.unloaded`, `plugin.error` |
| 生成事件 | 生成流程 | `generation.started`, `generation.progress`, `generation.completed` |
| 验证事件 | 质量验证 | `validation.completed` |
| 用户事件 | 用户操作 | `user.generation_requested` |
| 配置事件 | 配置变更 | `config.changed` |

### 3.2 插件接口标准

#### 3.2.1 基础插件接口 (BasePlugin)

```python
# core/plugin_interface.py
from abc import ABC, abstractmethod
from typing import Dict, Any, List, Optional
from dataclasses import dataclass, field
from enum import Enum

class PluginType(Enum):
    """插件类型枚举"""
    PROTOCOL = "protocol"
    AI = "ai"
    STORAGE = "storage"
    ANALYZER = "analyzer"
    GENERATOR = "generator"
    VALIDATOR = "validator"
    TOOL = "tool"

@dataclass
class PluginMetadata:
    """插件元数据"""
    id: str                          # 唯一标识符（如：outline-parser-v3）
    name: str                        # 显示名称
    version: str                     # 版本号（语义化版本）
    description: str                 # 描述
    author: str                      # 作者
    plugin_type: PluginType          # 插件类型
    api_version: str = "1.0"         # API版本
    priority: int = 100              # 加载优先级（越小越先）
    enabled: bool = True             # 是否启用
    dependencies: List[str] = field(default_factory=list)  # 依赖插件ID列表
    conflicts: List[str] = field(default_factory=list)     # 冲突插件ID列表
    permissions: List[str] = field(default_factory=list)   # 所需权限
    min_platform_version: str = "6.0.0"  # 最低平台版本

class BasePlugin(ABC):
    """插件基类 - 所有插件必须继承
    
    重要：BasePlugin不包含execute()方法，使用专用接口
    - AnalyzerPlugin.analyze()
    - GeneratorPlugin.generate()
    - ValidatorPlugin.validate()
    """

    @classmethod
    @abstractmethod
    def get_metadata(cls) -> PluginMetadata:
        """获取插件元数据"""
        pass

    @abstractmethod
    def initialize(self, context: 'PluginContext') -> bool:
        """初始化插件"""
        pass

    def shutdown(self) -> bool:
        """关闭插件 - 优雅关闭，释放资源"""
        return True
    
    def cleanup(self) -> bool:
        """强制资源回收 - 与shutdown区分"""
        return True
```

> **V1.2修正**：BasePlugin不包含execute()方法，使用专用接口（Analyzer/Generator/Validator等）

#### 3.2.2 Pydantic数据模型

```python
# core/models.py
from pydantic import BaseModel, Field, ConfigDict, field_validator
from typing import Dict, Any, List, Optional
from datetime import datetime, timezone

class GenerationRequest(BaseModel):
    """生成请求模型"""
    model_config = ConfigDict(from_attributes=True)

    chapter_title: str = Field(..., description="章节标题")
    chapter_outline: str = Field(..., description="章节大纲")
    word_count_target: int = Field(default=2000, ge=100, le=10000)
    max_iterations: int = Field(default=5, ge=1, le=10)
    score_threshold: float = Field(default=0.8, ge=0.0, le=1.0)

class GenerationResult(BaseModel):
    """生成结果模型"""
    model_config = ConfigDict(from_attributes=True)

    content: str = Field(..., description="生成内容")
    word_count: int = Field(..., description="实际字数")
    iteration_count: int = Field(..., description="迭代次数")
    final_score: float = Field(..., ge=0.0, le=1.0, description="最终评分")
    scores: Dict[str, float] = Field(..., description="各维度评分")
    timestamp: datetime = Field(
        default_factory=lambda: datetime.now(timezone.utc),
        description="生成时间"
    )

class ValidationScores(BaseModel):
    """验证评分模型"""
    model_config = ConfigDict(from_attributes=True)

    word_count_score: float = Field(ge=0.0, le=1.0)
    outline_score: float = Field(ge=0.0, le=1.0)
    style_score: float = Field(ge=0.0, le=1.0)
    character_score: float = Field(ge=0.0, le=1.0)
    worldview_score: float = Field(ge=0.0, le=1.0)
    naturalness_score: float = Field(ge=0.0, le=1.0)

    @property
    def total_score(self) -> float:
        """计算总评分（6维度加权）"""
        return (
            self.word_count_score * 0.10 +
            self.outline_score * 0.15 +
            self.style_score * 0.25 +
            self.character_score * 0.25 +
            self.worldview_score * 0.20 +
            self.naturalness_score * 0.05
        )
```

### 4.2 EventBus（事件总线）

**核心功能**：插件间发布-订阅通信，线程安全实现

**关键接口**：

| 方法 | 说明 | 参数 | 返回值 |
|------|------|------|--------|
| subscribe | 订阅事件 | event_type, handler, handler_id, priority, is_async, timeout | 订阅ID |
| unsubscribe | 取消订阅 | subscription_id | bool |
| publish | 异步发布 | event_type, data, source | None |
| publish_sync | 同步发布 | event_type, data, timeout | List[Any] |

**线程安全**：所有写操作使用 `threading.RLock`，读操作拷贝快照

**验证状态**：✅ 已验证（2026-03-21）

### 6.1 Agent系统设计与实现经验

> **经验来源**：Agent系统详细设计文档V2.0开发、自检与评审（2026-03-21）

#### 6.1.1 MasterAgent调度算法实现要点

**优先级队列设计**：
- 使用Python `heapq` 实现多级优先级队列，时间复杂度O(log n)
- 优先级定义：CRITICAL(0) > HIGH(1) > NORMAL(2) > LOW(3) > BACKGROUND(4)
- 同优先级任务按创建时间FIFO排序，确保公平性

**依赖解析实现**：
- 采用Kahn算法进行拓扑排序，检测循环依赖
- 依赖图必须是有向无环图（DAG），否则拒绝加载
- 依赖解析时机：任务提交时预检查，执行前最终确认

**重试机制设计**：
- 指数退避策略：`delay = base_delay * (2 ^ attempt) + jitter`
- jitter随机范围：0-20%，避免 thundering herd 问题
- 重试策略分级：FIXED（固定间隔）、EXPONENTIAL（指数退避）、IMMEDIATE（立即重试）

#### 6.1.2 Agent执行约束与熔断器

**Agent执行约束配置**（与架构文档一致）：

| Agent类型 | 超时时间 | 最大重试 | 回退策略 |
|-----------|----------|----------|----------|
| Thinker | 30s | 1次 | 使用缓存结果 |
| Optimizer | 20s | 1次 | 简化优化 |
| Validator | 15s | 0次 | 跳过验证 |
| Planner | 10s | 0次 | 使用默认计划 |

**熔断器状态机实现**：
```
CLOSED（正常）→ 失败次数≥阈值 → OPEN（熔断）
  ↑                                    ↓
  └──── 半开状态测试成功 ←── HALF_OPEN（半开）
```

- CLOSED状态：正常执行，记录失败次数
- OPEN状态：拒绝执行，直接返回降级结果
- HALF_OPEN状态：允许部分请求测试服务恢复

**四级容错机制**：
1. **重试层**：指数退避 + jitter，最多3次
2. **熔断层**：失败率>50%且次数≥5时熔断30秒
3. **模型回退**：DeepSeek失败→OpenAI→Anthropic
4. **缓存降级**：返回最近一次成功结果

#### 解决方案

**修改`core/plugin_registry.py`的`register()`方法**：
```python
def register(
    self,
    plugin_id: str,
    metadata: Union[PydanticPluginMetadata, Any],  # 支持Pydantic和dataclass两种类型
    instance: Optional[Any] = None
) -> bool:
    # V1.4新增：处理不同类型的metadata
    if isinstance(metadata, PydanticPluginMetadata):
        pydantic_metadata = metadata
    elif hasattr(metadata, 'to_dict'):
        # dataclass版本，转换为Pydantic
        data = metadata.to_dict()
        pydantic_metadata = PydanticPluginMetadata(**data)
    elif isinstance(metadata, dict):
        pydantic_metadata = PydanticPluginMetadata(**metadata)
    else:
        # 尝试从对象属性创建
        pydantic_metadata = PydanticPluginMetadata(
            id=getattr(metadata, 'id', plugin_id),
            name=getattr(metadata, 'name', plugin_id),
            # ...其他字段

_ALLOWED_TRANSITIONS: Dict[PluginState, Set[PluginState]] = {

## 九、Git工作流上传经验（V3.4）

### 9.1 任务概述

完成GitHub远程仓库建立、分支模型配置、CI工作流设置和完整项目上传。

### 9.2 关键经验

#### 9.2.1 网络代理配置

- **问题**：国内网络直连GitHub HTTPS端口（443）超时
- **原因**：系统代理`127.0.0.1:7897`未同步到Git配置
- **解决**：
  ```bash
  # 检测系统代理
  Get-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings" | Select-Object ProxyEnable, ProxyServer
  
  # 配置Git代理
  git config --global http.proxy http://127.0.0.1:7897
  git config --global https.proxy http://127.0.0.1:7897
  
  # 或使用环境变量（推荐）
  $env:HTTP_PROXY="http://127.0.0.1:7897"
  $env:HTTPS_PROXY="http://127.0.0.1:7897"
  ```

#### 9.2.2 GitHub CLI认证权限

- **问题**：推送workflow文件时报错`refusing to allow an OAuth App to create or update workflow without workflow scope`
- **原因**：默认认证缺少workflow权限
- **解决**：
  ```bash
  # 刷新认证并添加workflow权限
  gh auth refresh -h github.com -s workflow,repo
  ```

#### 9.2.3 分支保护与PR合并

- **问题**：main分支保护后无法直接推送
- **流程**：
  1. 从develop分支开发
  2. 推送到origin/develop
  3. 创建PR：`gh pr create --base main --head develop`
  4. 等待CI通过
  5. 合并PR：`gh pr merge 1 --squash`
- **注意事项**：
  - main分支保护后，必须通过PR合并
  - PR需要CI检查通过才能合并
  - 可以临时取消保护，合并后恢复

#### 9.2.4 CI检查失败处理

- **Black格式检查失败**：
  ```bash
  # 本地预格式化
  black . --exclude "经验文档|__pycache__|.git|.github|.workbuddy"
  ```
- **Flake8代码检查失败**：
  - 常见错误：`F821 undefined name`（缺少导入）
  - 解决：添加缺失的`import`语句
