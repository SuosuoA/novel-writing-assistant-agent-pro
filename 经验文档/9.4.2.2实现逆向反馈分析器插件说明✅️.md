### 3.2 核心方法

#### 3.2.1 analyze_chapter_vs_settings

**功能**: 分析单章节与设定的冲突

**参数**:
- `chapter_text`: 章节文本内容
- `current_settings`: 当前项目设定
  - `outline`: 大纲文本
  - `characters`: 人物设定列表
  - `worldview`: 世界观设定文本
- `chapter_id`: 章节ID或标题

**返回**: `ConsistencyReport`

**流程**:
```
输入章节文本和当前设定
    ↓
检查缓存（有则返回）
    ↓
基础规则分析（快速）
    - 人物姓名检查
    - 世界观术语检查
    - 大纲顺序检查
    ↓
深度语义分析（AI）
    - 构建提示词
    - 调用LLM
    - 解析分析结果
    ↓
生成分析摘要
    ↓
缓存结果
    ↓
返回 ConsistencyReport
```

#### 3.2.2 analyze_project

**功能**: 分析整个项目的一致性

**参数**:
- `project_data`: 完整项目数据
  - `project_name`: 项目名称
  - `chapters`: 章节列表
  - `outline`: 大纲文本
  - `characters`: 人物设定列表
  - `worldview`: 世界观设定文本
- `options`: 分析选项

**返回**: `ConsistencyReport`（综合报告）

**流程**:
```
提取项目设定
    ↓
遍历所有章节
    ↓
对每个章节调用 analyze_chapter_vs_settings
    ↓
合并所有冲突项到综合报告
    ↓
生成综合摘要
    ↓
返回 ConsistencyReport
```

#### 3.2.3 generate_corrections

**功能**: 根据冲突报告生成修正后的设定

**参数**:
- `report`: 一致性分析报告
- `current_settings`: 当前项目设定
- `options`: 修正选项
  - `auto_fix_low`: 是否自动修正低优先级冲突
  - `preserve_original`: 是否保留原始设定备份

**返回**: 
```python
{
    "updated_outline": str,
    "updated_characters": List[Dict],
    "updated_worldview": str,
    "suggestions": List[str],
    "backup": Optional[Dict]
}
```

### 3.3 事件订阅

**订阅事件**:
- `chapter.generated`: 章节生成后自动进行一致性分析

**发布事件**:
- `consistency.issue.detected`: 检测到设定冲突时发布

**异步分析流程**:
```python
def _on_chapter_generated(self, event):
    """处理章节生成事件"""
    chapter_data = event.data
    
    # 异步分析,避免阻塞主流程
    thread = threading.Thread(
        target=self._async_analyze_chapter,
        args=(chapter_data,),
        daemon=True
    )
    thread.start()
```

## 四、数据模型

### 4.1 ConsistencyIssue（冲突项）

**字段**:
| 字段 | 类型 | 说明 |
|------|------|------|
| `issue_id` | str | 冲突项唯一标识 |
| `issue_type` | ConsistencyIssueType | 冲突类型（outline/character/worldview） |
| `severity` | ConsistencySeverity | 严重程度（low/medium/high） |
| `description` | str | 冲突描述 |
| `suggested_fix` | str | 建议修正方案 |
| `original_content` | str | 原始设定内容 |
| `chapter_reference` | str | 引发冲突的章节ID或标题 |
| `detected_at` | float | 检测时间戳 |
| `element_name` | str | 冲突涉及的元素名称 |
| `confidence` | float | 检测置信度（0.0-1.0） |

**示例**:

## 七、性能优化

### 7.1 缓存策略

- **章节级缓存**: 每个章节的分析结果独立缓存
- **缓存键**: `chapter_id`
- **缓存值**: `ConsistencyReport` 对象
- **过期策略**: 手动清除,无自动过期

**缓存效果**:
- 首次分析: 需要调用LLM,耗时约 3-5 秒
- 缓存命中: 直接返回,耗时 < 10 毫秒

### 7.2 异步分析

- **事件驱动**: 订阅 `chapter.generated` 事件
- **异步执行**: 在独立线程中执行分析
- **非阻塞**: 不影响章节生成流程

### 7.3 内容截断

- **提示词长度限制**: 章节内容截断至 2000 字符
- **设定内容限制**: 各设定项截断至 500 字符
- **原因**: 避免Token超限,控制API调用成本

### 10.1 单章节一致性分析

```python
from core.plugin_interface import ReverseFeedbackAnalyzerPlugin

plugin = plugin_registry.get_plugin("reverse-feedback-analyzer")
```

## 十一、已知限制

### 11.1 功能限制

1. **LLM质量依赖**: 深度语义分析依赖LLM的理解能力,不同模型可能产生不同结果
2. **Token限制**: 长章节会被截断,可能遗漏部分冲突
3. **主观性**: 性格偏离等判断具有一定主观性
4. **缓存失效**: 设定变更后需要手动清除缓存重新分析
5. **依赖插件**: 部分功能依赖其他插件,依赖不可用时功能受限

### 11.2 性能限制

1. **AI分析耗时**: 首次分析需要 3-5 秒（依赖LLM响应速度）
2. **批量分析耗时**: 项目整体分析时间与章节数量成正比
3. **内存占用**: 缓存大量分析结果会占用内存

3. **异步分析**: 事件驱动 + 异步执行,不阻塞主流程
4. **降级策略**: LLM失败时返回基础规则分析结果,保证可用性
5. **缓存优化**: 章节级缓存,避免重复调用AI
