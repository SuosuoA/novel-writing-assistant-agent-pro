## 二、技术架构

### 2.1 WAL协议核心原理

**Write-Ahead Log（WAL）**是数据库系统常用的技术，核心原则：**在执行操作前，先记录操作日志**。

在OpenClaw中的应用：
```
AI调用流程：
1. 用户请求生成第3章
2. ✅ WAL写入：记录"即将生成第3章"（同步落盘）
3. 调用LLM API
4. ❓ 可能崩溃在这里（程序崩溃/断电/网络中断）
5. ✅ WAL标记完成："生成第3章成功"
```

如果步骤4崩溃，下次启动时可以从WAL恢复：
```
启动流程：
1. 检查WAL文件
2. 发现pending状态的记录："即将生成第3章"
3. 提示用户："上次未完成的操作：生成第3章，是否继续？"
```

### 2.2 核心数据模型

**WALRecord（WAL记录）**:
```python
class WALRecord(BaseModel):
    record_id: str              # 记录ID（时间戳）
    timestamp: str              # 记录时间
    operation: str              # 操作描述（如："生成第3章"）
    task_id: Optional[str]      # 任务ID
    context: Dict[str, Any]     # 上下文数据（章节号、大纲、人物等）
    status: str                 # 状态：pending/completed/failed
    error: Optional[str]        # 错误信息
```

**WALState（WAL状态）**:
```python
class WALState(BaseModel):
    current_record: Optional[WALRecord]  # 当前WAL记录
    last_write_time: Optional[str]       # 上次写入时间
    total_writes: int                    # 总写入次数
    failed_writes: int                   # 失败写入次数
```

### 2.3 核心方法

**write_before_ai_call（WAL协议核心）**:
```python
def write_before_ai_call(
    self,
    context: Dict[str, Any],
    operation: str = "",
    task_id: Optional[str] = None
) -> bool:
    """
    WAL协议：在AI调用前写入状态（同步阻塞式）
    
    关键步骤：
    1. 创建WAL记录（status="pending"）
    2. 同步写入文件（fsync确保落盘）
    3. 发布事件通知监控系统

        bool: 写入是否成功（失败则不应继续调用AI）
    """
```

**mark_completed / mark_failed**:
```python
def mark_completed(self, result: Optional[Dict] = None):
    """标记当前WAL记录为已完成"""

    """标记当前WAL记录为失败"""
```

**recover_on_startup**:
```python
def recover_on_startup() -> Optional[Dict[str, Any]]:
    """
    启动时恢复上次会话状态
    
    Returns:
        如果有pending状态的记录，返回恢复数据
    """
```

## 三、关键技术实现

### 3.1 同步写入（fsync落盘）

**核心要求**：确保数据真正写入磁盘，而不是停留在操作系统缓冲区。

```python
def _write_state_to_disk(self) -> None:
    """同步写入状态到磁盘（阻塞式）"""
    # 写入临时文件（原子化）
    temp_file = self.wal_file.with_suffix('.tmp')
    with open(temp_file, 'w', encoding='utf-8') as f:
        json.dump(state_data, f, ensure_ascii=False, indent=2)
        f.flush()  # 刷新Python缓冲区
        # 关键！fsync确保落盘
        import os
        os.fsync(f.fileno())
    
    # 原子化重命名
    temp_file.replace(self.wal_file)
```

**为什么需要fsync？**
- `write()` 只是把数据写入操作系统缓冲区
- `flush()` 只是把Python缓冲区的数据刷新到OS缓冲区
- `fsync()` 才是真正把数据写入物理磁盘

### 3.2 原子化写入

**问题**：如果在写入过程中崩溃，可能导致文件损坏。

**解决方案**：
1. 先写入临时文件（`.tmp`）
2. 确保临时文件完整写入磁盘（fsync）
3. 原子化重命名（`replace`）

```python
temp_file = self.wal_file.with_suffix('.tmp')
# 写入临时文件...
temp_file.replace(self.wal_file)  # 原子操作
```

### 3.3 线程安全

**设计理念**：WAL记录**当前单个AI调用**，不支持并发操作。

**原因**：
- WAL的目的是记录"当前正在做什么"，以便崩溃后恢复
- 如果允许多个并发WAL记录，崩溃后无法确定哪个是真正的"当前操作"
- 真实场景中，AI调用通常是串行的（有锁保护）

**实现**：
```python
class WALManager:
    def __init__(self, workspace: Path):
        # 线程锁
        self._write_lock = threading.RLock()  # 写入锁
        self._state_lock = threading.RLock()  # 状态锁
```

### 3.4 EventBus集成

**发布的事件**：
| 事件类型 | 说明 | 数据 |
|---------|------|------|
| `wal.write.success` | WAL写入成功 | record_id, operation, timestamp |
| `wal.write.failed` | WAL写入失败 | error, operation |
| `wal.completed` | AI调用完成 | record_id, operation |
| `wal.failed` | AI调用失败 | record_id, operation, error |
| `wal.recovery.needed` | 发现待恢复记录 | record |
| `wal.cleared` | WAL清空 | {} |

**延迟导入避免循环依赖**：
```python
def _get_event_bus(self):
    """延迟获取EventBus实例"""
    if self._event_bus is None:
        try:
            from .event_bus import get_event_bus
            self._event_bus = get_event_bus()
        except ImportError:
            self._event_bus = None
    return self._event_bus
```

### 3.5 单例工厂模式

**设计理念**：全局唯一实例，确保状态一致性。

```python
_wal_manager_instance: Optional[WALManager] = None
_wal_manager_lock = threading.Lock()

def get_wal_manager(workspace: Optional[Path] = None) -> WALManager:
    """获取WAL管理器单例实例"""
    global _wal_manager_instance
    
    if _wal_manager_instance is None:
        with _wal_manager_lock:
            if _wal_manager_instance is None:
                if workspace is None:
                    workspace = Path.cwd()
                _wal_manager_instance = WALManager(workspace)
    
    return _wal_manager_instance

def reset_wal_manager() -> None:
    """重置WAL管理器单例（仅用于测试）"""
```

```python
    context={
```

## 七、架构意义

### 7.1 在OpenClaw mem9中的位置

WAL协议是OpenClaw mem9的**核心机制**，与SESSION-STATE协同工作：

```
OpenClaw mem9架构：
┌─────────────────────────────────────┐
│  L1: SESSION-STATE.md（热记忆）      │  ← 当前会话状态
├─────────────────────────────────────┤
│  WAL协议（核心机制）                 │  ← 本次实现
│  - AI调用前写入状态                  │
│  - 确保中断可恢复                    │
│  - fsync落盘保证                     │
├─────────────────────────────────────┤
│  L2: LanceDB向量库（温记忆）         │  ← Sprint 1-2后续
│  L3: Git-Notes（冷记忆）             │  ← Sprint 1-2后续
│  L4: MEMORY.md（精选档案）           │  ← 长期维护
│  L5: 云备份（异地备份）              │  ← Sprint 11-12
└─────────────────────────────────────┘
```

### 7.2 解决的核心问题

**问题**: AI调用中途崩溃导致数据丢失

**场景示例**:
```
用户场景：
1. 用户请求生成长篇小说第10章
2. 系统调用LLM API（可能需要10-30秒）
3. 此时程序崩溃（断电/系统更新/内存溢出）
4. 用户重启程序，发现之前的大纲设定、人物信息全部丢失
```

**WAL解决方案**:
```
WAL流程：
1. 用户请求生成第10章
2. ✅ WAL写入：
   - 操作：生成第10章
   - 上下文：{章节号:10, 大纲:..., 人物:[张三,李四]}
   - 状态：pending
   - fsync确保落盘
3. 调用LLM API...
4. ❌ 崩溃

恢复流程：
1. 用户重启程序
2. WAL检查：发现pending记录"生成第10章"
3. 提示用户："检测到上次未完成的操作，是否继续？"
4. 用户确认后恢复上下文（大纲、人物）继续执行
```

### 7.3 对比传统方案

| 方案 | 数据安全性 | 性能影响 | 实现复杂度 |
|------|-----------|---------|-----------|
| 无保护 | ❌ 崩溃即丢失 | ✅ 无开销 | ✅ 简单 |
| 定期保存 | ⚠️ 可能丢失部分数据 | ⚠️ 定期IO | ⚠️ 中等 |
| **WAL协议** | ✅ 100%不丢失 | ⚠️ 每次AI调用前IO | ⚠️ 中等 |

**WAL的优势**：
- 数据安全性最高：每次AI调用前都持久化
- 开销可控：只在AI调用前写入（频率低）
- 恢复简单：只需读取WAL文件

## 九、性能考量

### 9.1 性能影响

**WAL写入开销**:
- 文件写入：~1-2ms
- fsync落盘：~5-10ms
- 总开销：~10ms

**对比AI调用**:
- AI调用通常需要：500-5000ms
- WAL开销占比：~0.2-2%

**结论**: 性能影响可忽略不计。

### 9.2 优化方向

如果未来需要优化性能（如高频AI调用场景）：

1. **批量写入**：累积多个WAL记录后一次性写入
2. **异步fsync**：先写入缓冲区，后台线程定期fsync
3. **WAL轮转**：限制WAL文件大小，定期归档

**当前不需要**：小说创作场景下，AI调用频率低（每分钟1-2次），WAL开销可忽略。

## 十、注意事项

### 10.1 使用限制

1. **不支持并发**：WAL记录当前单个AI调用，不支持多线程并发写入
2. **手动标记完成**：AI调用成功后必须调用`mark_completed()`
3. **定期清理**：长期运行后应定期清理已完成的WAL记录

### 10.2 最佳实践

1. **AI调用前必须写入WAL**：
   ```python
   # ✅ 正确
   wal_manager.write_before_ai_call(...)
   result = llm_api.generate(...)
   
   # ❌ 错误
   result = llm_api.generate(...)  # 直接调用AI，无保护
   ```

2. **异常处理**：
   ```python
   try:
       result = llm_api.generate(...)
       wal_manager.mark_completed()
   except Exception as e:
       wal_manager.mark_failed(str(e))
       raise
   ```

3. **启动时检查恢复**：
   ```python
   recovered = wal_manager.recover_on_startup()
   if recovered:
       # 询问用户是否继续
       ...
   ```
