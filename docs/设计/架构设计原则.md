## 一、架构愿景

### 1.1 设计理念

**微内核架构**：核心系统仅提供基础调度服务，所有业务功能以插件形式存在。

**核心价值**：
1. **高内聚低耦合**：模块间依赖最小化，循环依赖=0
2. **热插拔支持**：插件动态加载/卸载，开发环境支持热重载
3. **可扩展性**：新增功能无需修改核心代码
4. **可维护性**：清晰的模块边界，便于测试和调试

### 1.2 架构分层


## 二、核心设计原则

### 2.1 单一职责原则 (SRP)

**定义**：每个模块只负责一项职责。

**实践**：
- **EventBus**：仅负责事件发布订阅
- **PluginRegistry**：仅负责插件注册和状态管理
- **ServiceLocator**：仅负责服务依赖注入
- **ConfigManager**：仅负责配置读写

**反例**：
```python
# ❌ 错误：EventManager既负责事件发布，又负责插件管理
class EventManager:
    def publish_event(self, event): ...
    def register_plugin(self, plugin): ...  # 职责不单一
```

**正例**：
```python
# ✅ 正确：职责分离
class EventBus:
    def publish(self, event_type, data): ...
    def subscribe(self, event_type, callback): ...

class PluginRegistry:
    def register(self, plugin): ...
    def get_plugin(self, plugin_id): ...
```

### 2.2 开闭原则 (OCP)

**定义**：对扩展开放，对修改关闭。

**实践**：
- 新增功能：创建新插件，无需修改核心代码
- 新增Agent：创建Agent适配器，无需修改MasterAgent
- 新增验证规则：实现ValidatorPlugin接口

**示例**：
```python
# 新增世界观验证插件（不修改核心代码）
class WorldviewValidatorPlugin(ValidatorPlugin):
    def validate(self, content, context):
        # 世界观验证逻辑
        return ValidationResult(...)

# 插件系统自动发现和加载
plugin_loader.discover_plugins()
```

### 2.3 依赖倒置原则 (DIP)

**定义**：高层模块不依赖低层模块，两者都依赖抽象。

**实践**：
- **高层模块**：MasterAgent、PipelineOrchestrator
- **抽象接口**：BasePlugin、BaseAgent
- **低层模块**：具体插件实现、Agent实现

**示例**：
```python
# ✅ 正确：依赖抽象接口
class MasterAgent:
    def __init__(self, registry: PluginRegistry):
        self._registry = registry  # 依赖抽象
    
    def execute_plugin(self, plugin_id: str):
        plugin = self._registry.get_plugin(plugin_id)
        return plugin.execute(...)

# ❌ 错误：依赖具体实现
class MasterAgent:
    def __init__(self):
        self._outline_parser = OutlineParserPlugin()  # 依赖具体实现
```

### 2.4 接口隔离原则 (ISP)

**定义**：客户端不应该依赖它不需要的接口。

**实践**：
- **通用接口**：BasePlugin（所有插件通用）
- **专用接口**：
  - AnalyzerPlugin（分析器插件）
  - GeneratorPlugin（生成器插件）
  - ValidatorPlugin（验证器插件）

**示例**：
```python
# ✅ 正确：接口隔离
class AnalyzerPlugin(BasePlugin):
    def analyze(self, content, context) -> AnalysisResult: ...

class GeneratorPlugin(BasePlugin):
    def generate(self, request) -> GenerationResult: ...

class ValidatorPlugin(BasePlugin):
    def validate(self, content, context) -> ValidationResult: ...

# ❌ 错误：臃肿接口
class PluginInterface:
    def analyze(self): ...      # Generator不需要
    def generate(self): ...     # Validator不需要
    def validate(self): ...     # Analyzer不需要
```

### 2.5 里氏替换原则 (LSP)

**定义**：子类可以替换父类而不影响程序正确性。

**实践**：
- 所有插件实现都可以替换BasePlugin
- 所有Agent实现都可以替换BaseAgent
- 替换后核心功能正常运行

**示例**：
```python
# ✅ 正确：所有插件都可以替换BasePlugin
def execute_any_plugin(plugin: BasePlugin):
    result = plugin.execute(...)
    assert result.success is not None

# 所有插件实现都兼容
outline_parser = OutlineParserPlugin()
validator = QualityValidatorPlugin()
execute_any_plugin(outline_parser)  # OK
execute_any_plugin(validator)       # OK
```

## 三、微内核架构设计

### 3.1 核心组件

**EventBus（事件总线）**：
- **职责**：事件发布订阅，解耦模块通信
- **特性**：
  - ThreadPoolExecutor真异步执行
  - 线程安全（RLock保护）
  - 异常隔离（观察者异常不影响其他观察者）
  - 事件类型：`{prefix}.{action}`格式

**PluginRegistry（插件注册表）**：
- **职责**：插件注册、状态管理、生命周期控制
- **特性**：
  - 5状态模型：UNLOADED → LOADED → ACTIVE → UNLOADING → ERROR
  - V5保护模块：禁止卸载/重载9个核心模块
  - 依赖解析：拓扑排序 + 循环检测

**ServiceLocator（服务定位器）**：
- **职责**：依赖注入容器，服务生命周期管理
- **特性**：
  - 单例管理：全局唯一服务实例
  - 延迟初始化：首次使用时创建
  - 循环依赖检测：拓扑排序 + 异常抛出

**ConfigManager（配置管理器）**：
- **职责**：配置读写、热重载、环境变量支持
- **特性**：
  - YAML格式配置文件
  - 配置变更事件：`config.changed`
  - 环境变量覆盖：`NOVEL_ASSISTANT_*`

### 3.2 启动流程

```python
# 启动流程（分层延迟加载）
1. 核心层（同步，必须成功）
   - ConfigManager.load()
   - EventBus.initialize()
   - ServiceLocator.register_services()
   - PluginRegistry.initialize()

2. 插件层（异步，可热重载）
   - PluginLoader.discover_plugins()
   - PluginLoader.load_protected_plugins()  # V5核心模块
   - PluginLoader.load_optional_plugins()

3. Agent层（延迟加载）
   - MasterAgent.initialize()
   - AgentPool.register_builtin_agents()

4. UI层（主线程）
   - MainWindow.initialize()
   - UI组件延迟创建
```

## 四、插件系统设计

### 4.1 插件接口

**BasePlugin（基础接口）**：
```python
class BasePlugin(ABC):
    @property
    @abstractmethod
    def plugin_id(self) -> str: ...
    
    @property
    @abstractmethod
    def plugin_type(self) -> PluginType: ...
    
    @abstractmethod
    def initialize(self, context: PluginContext) -> bool: ...
    
    @abstractmethod
    def execute(self, operation: str, payload: Dict[str, Any]) -> Any: ...
    
    def shutdown(self) -> bool: ...
```

**专用接口**：
- **AnalyzerPlugin**：分析内容，返回分析结果
- **GeneratorPlugin**：生成内容，支持流式生成
- **ValidatorPlugin**：验证内容，返回评分
- **ToolPlugin**：工具操作，如文件导入导出

### 4.2 插件类型（7种）

| 类型 | 职责 | 示例插件 |
|------|------|----------|
| Protocol | 协议支持 | HTTP、WebSocket |
| AI | AI服务封装 | DeepSeek、OpenAI、Local |
| Storage | 存储服务 | SQLite、LanceDB、文件系统 |
| Analyzer | 内容分析 | 大纲解析、风格学习、世界观解析 |
| Generator | 内容生成 | 章节生成、续写生成、快捷创作 |
| Validator | 质量验证 | 6维度评分、一致性检查 |
| Tool | 工具操作 | 文件导入导出、项目管理 |

### 4.3 插件生命周期

```
UNLOADED（未加载）
    ↓ load()
LOADED（已加载）
    ↓ initialize()
ACTIVE（激活）
    ↓ shutdown()
UNLOADING（卸载中）
    ↓
UNLOADED（未加载）

ERROR（错误状态，可恢复到LOADED）
```

### 4.4 插件权限分级

| 等级 | 类型 | 加载 | 重载 | 卸载 | 进程隔离 |
|------|------|------|------|------|----------|
| L0 | 官方插件 | ✅ | ✅ | ✅ | ❌ |
| L1 | 第三方签名 | ✅ | ✅ | ❌ | ❌ |
| L2 | 未知来源 | ✅ | ❌ | ❌ | ✅ |
| L3 | 风险插件 | ❌ | ❌ | ❌ | ❌ |

## 五、Agent系统设计

### 5.1 Agent架构

```
MasterAgent（总控Agent）
├─ AgentRegistry（Agent注册表）
├─ TaskQueue（优先级任务队列）
├─ DependencyResolver（依赖解析器）
├─ ThreadPoolExecutor（线程池执行器）
└─ AgentPool（Agent实例池）
    ├─ OutlineAnalysisAgent（大纲分析Agent）
    ├─ StyleLearningAgent（风格学习Agent）
    ├─ NovelGenerationAgent（内容生成Agent）
    └─ QualityValidationAgent（质量验证Agent）
```

### 5.2 Agent协作模式

**链式模式**：
```
OutlineAnalysisAgent → StyleLearningAgent → NovelGenerationAgent → QualityValidationAgent
```

**树状模式**：
```
                MasterAgent
               /     |      \
          Agent1   Agent2   Agent3
           /  \               |
       Agent4 Agent5        Agent6
```

**网状模式**：
```
Agent1 ←→ Agent2
  ↕        ↕
Agent3 ←→ Agent4
```

### 5.3 Agent与插件的关系

**适配器模式**：
- Agent作为插件适配器，不修改插件代码
- `execute()`方法将插件输出转换为AgentResult
- 通过ServiceLocator获取PluginRegistry实例

**示例**：
```python
class OutlineAnalysisAgent(BaseAgent):
    PLUGIN_ID = "outline-parser-v3"
    
    def execute(self, task_id, payload, context):
        # 获取插件
        registry = self.get_service(PluginRegistry)
        plugin = registry.get_plugin(self.PLUGIN_ID)
        
        # 调用插件
        result = plugin.execute("parse", payload)
        
        # 转换为AgentResult
        return AgentResult(
            success=result.success,
            data=result.data,
            error=result.error
        )
```

## 六、长期记忆架构（OpenClaw mem9）

### 6.1 五层记忆架构

| 层级 | 名称 | 实现 | 特性 |
|:---:|:---|:---|:---|
| L1 | 热记忆 | SESSION-STATE.md | 当前会话状态，WAL协议保护 |
| WAL | 原子化写入 | wal.json | AI响应前持久化，中断可恢复 |
| L2 | 温记忆 | LanceDB向量库 | 章节向量、知识向量、风格向量 |
| L3 | 冷记忆 | Git-Notes | 分支感知、决策记录、历史版本 |
| L4 | 精选档案 | MEMORY.md | 每日冥想精简，保留关键决策 |

### 6.2 WAL协议

**核心机制**：
1. AI调用前，先写入状态到WAL文件
2. AI调用成功，标记WAL为完成
3. AI调用失败，标记WAL为失败
4. 程序启动时，恢复未完成的WAL记录

**实现**：
```python
# AI调用前
wal_manager.write_before_ai_call({
    "operation": "generate_chapter",
    "chapter_number": 3,
    "context": {...}
})

# AI调用
try:
    result = llm_client.generate(...)
    wal_manager.mark_completed(result)
except Exception as e:
    wal_manager.mark_failed(str(e))
```

### 6.3 向量检索架构

**向量类型**：
- **章节向量**：长篇连贯性，召回相似章节
- **知识向量**：评分反馈，召回相关知识
- **风格向量**：风格学习，召回相似风格

**检索流程**：
```
1. 识别题材（科幻/玄幻/历史/通用）
2. 加载对应分类知识库
3. 向量召回 top-k 相似内容
4. 相似度过滤（score ≥ 0.6）
5. 返回召回结果
```

## 八、性能优化设计

### 8.1 启动优化

**分层延迟加载**：
1. 核心层：同步加载，必须成功（<1秒）
2. 插件层：异步加载，可后台加载（<3秒）
3. Agent层：延迟加载，首次使用时创建
4. UI层：延迟创建，首次切换时创建

**启动时间目标**：
- 核心服务启动：<1秒
- 首次可用时间：<3秒
- 完全加载时间：<5秒

### 8.2 缓存策略

**四级缓存**：
1. **内存缓存**：LRU缓存，容量限制1000条
2. **TTL缓存**：带过期时间的缓存
3. **向量缓存**：LanceDB向量索引
4. **文件缓存**：热榜数据、风格档案

**缓存预热**：
- 用户登录时预加载常用缓存
- 世界观、大纲、人物、风格四类数据

### 8.3 并发优化

**线程池配置**：
- EventBus线程池：max_workers=10
- Agent执行线程池：max_workers=4
- 异步任务线程池：max_workers=5

**数据库并发**：
- SQLite WAL模式：支持并发读
- 连接池：最大连接数10

## 九、安全设计

### 9.1 API密钥保护

**保护措施**：
1. `config.yaml` 从 git 追踪中移除
2. `.gitignore` 添加保护规则
3. 提供 `config.yaml.example` 作为模板
4. 代码审查检查点：禁止密钥提交

### 9.2 输入验证

**Pydantic模型验证**：
```python
class GenerationRequest(BaseModel):
    chapter_title: str = Field(..., min_length=1, max_length=100)
    chapter_number: int = Field(..., ge=1, le=1000)
    word_count: int = Field(..., ge=1000, le=10000)
    
    @validator('chapter_title')
    def validate_title(cls, v):
        # 防止注入攻击
        if '<' in v or '>' in v:
            raise ValueError('Invalid characters in title')
        return v
```

### 9.3 异常处理

**异常分类**：
- **可恢复异常**：重试、降级
- **不可恢复异常**：记录日志、优雅退出

**异常处理原则**：
1. 捕获具体异常，避免空except
2. 记录完整异常信息（堆栈、上下文）
3. 发布异常事件到EventBus
4. 提供友好的用户提示

## 十、扩展性设计

### 10.1 新增插件

**步骤**：
1. 创建插件目录：`plugins/{plugin-id}/`
2. 实现插件接口：继承BasePlugin或专用接口
3. 编写plugin.json：元数据、依赖、权限
4. 放入plugins目录：自动发现和加载

### 10.2 新增Agent

**步骤**：
1. 创建Agent类：继承BaseAgent
2. 实现execute方法：调用插件或服务
3. 注册到AgentPool：`agent_pool.register_agent(agent)`
4. 配置到流水线：在PipelineOrchestrator中定义阶段

### 10.3 新增知识库

**步骤**：
1. 准备知识数据：JSON格式，符合Schema
2. 调用KnowledgeManager导入：`import_from_json()`
3. 自动生成向量嵌入：EmbeddingFunction
4. 支持向量检索：召回相关知识
